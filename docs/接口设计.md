### 0.1 基础
- **协议**:HTTPS;**风格**:RESTful;**数据格式**:JSON(`Content-Type: application/json`)。
- **通用请求头**:
  - `Authorization: Bearer <accessToken>`:登录态接口必带。
  - `X-Region: CN | EU`:请求区域。
  - `Accept-Language: zh-CN | en-US`:响应文案语言,默认 `zh-CN`。
- **Base URL**:
  - C 端(PC/APP 用户):`/api/c/v1`
  - B 端(商户端):`/api/b/v1`
  - 管理端(运营后台):`/api/admin/v1`
- 三类 API 同一套后端服务,按前缀 + 角色鉴权隔离(对应需求文档 §4 说明)。

### 0.2 统一响应体
```json
{
  "code": 0,           // 0 成功;非 0 为业务错误码
  "message": "预约已提交", // 当前语言展示文案
  "messageKey": "reservation.submit_success", // 稳定 i18n key
  "data": { },         // 业务数据,失败时为 null
  "traceId": "xxx"     // 链路追踪 ID(排查用)
}
```

### 0.3 分页约定
- 请求:`?page=1&pageSize=20`(page 从 1 开始),或游标 `?cursor=xxx&pageSize=20`(feed 流用)。
- 响应 `data`:
```json
{
  "list": [],
  "total": 100,        // 游标分页时可为 null
  "page": 1,
  "pageSize": 20,
  "hasMore": true
}
```

### 0.4 鉴权
- **JWT**:登录后返回 `accessToken`(短期,2h)+ `refreshToken`(长期,30d)。
- 请求头:`Authorization: Bearer <accessToken>`。
- **鉴权级别**(对应需求 §3.9 免登录浏览):
  - 🟢 **公开**:游客可访问,不带 token 也返回公开数据(首页、列表、详情、搜索、榜单、看点评、看社区)。
  - 🔴 **需登录**:必须带有效 token(写点评、点赞、收藏、下单、我的、私信等)。
  - 🔵 **需商户角色**:B 端接口。
  - 🟣 **需管理员角色 + RBAC 权限点**:管理端接口。
- token 失效返回 `code=401`,前端用 refreshToken 换新;refresh 也失效则拉起登录浮层。

### 0.5 区域(region)
- `X-Region` 表示 `request_region`;当前 `Web` 默认 `CN` 但已支持切换 `CN / EU`,`APP` 规划默认 `EU`。
- 业务表里的 `region` 表示 `data_region`,后端按它过滤商户/城市/点评/团购/社区数据。
- 用户资料里的 `preferredRegion` 仅表示账号默认偏好,不参与本次请求访问控制。
- 商户端一期按单区域经营设计;当前已落地的 `GET /api/b/v1/account/me`、`GET /api/b/v1/roles`、`GET /api/b/v1/shops` 要求 `X-Region` 与商户经营区域一致,错区请求直接返回 `401`。

### 0.6 通用错误码(节选)
| code | 含义 |
|---|---|
| 0 | 成功 |
| 400 | 参数错误 |
| 401 | 未登录 / token 失效 |
| 403 | 无权限 |
| 404 | 资源不存在 |
| 409 | 冲突(如重复操作) |
| 429 | 限流 |
| 500 | 服务器错误 |
| 1xxx | 用户模块业务错误 |
| 2xxx | 商户模块 |
| 3xxx | 点评模块 |
| 4xxx | 交易模块 |
| 5xxx | 社区模块 |

### 0.7 i18n 返回策略
- `message` 返回当前 `Accept-Language` 对应的展示文案。
- `messageKey` 返回稳定键,前端若要完全本地化,以它为准。
- 业务对象里的展示字段统一返回当前语言文案,例如 `statusText`、`roleName`、`typeText`。
- 未命中语言时回退 `zh-CN`;管理端默认 `zh-CN`,后续再扩多语言后台。

### 0.8 幂等约定
- 这些写接口必须带 `Idempotency-Key`: `POST /auth/send-code`、`POST /orders`、`POST /orders/:id/pay`、`POST /reservations`、`POST /reservations/:id/reschedule`、`POST /devices/register`、B 端预订动作接口、`POST /privacy/export-tasks`、`POST /privacy/delete-tasks`。
- `Idempotency-Key` 建议客户端生成 UUIDv4 或等价随机串,长度 `16-64`。
- 服务端幂等作用域 = `调用方身份 + 方法 + 路由模板 + key`;同 key + 同请求体返回首个处理结果,同 key + 不同请求体直接 `409`。
- 默认保留时间:
  - 普通写接口: `24h`
  - 验证码发送: `10m`
  - 支付回调去重: `7d`
- 当前仓库已把 `Idempotency-Key` 接入带 key 的 JSON 写请求;默认本地内存,配置 `APP_STATE_STORE_PROVIDER=redis` 后可跨实例共享。MySQL 持久化审计和支付回调去重还没接,别把当前版本硬吹成支付级生产幂等。

### 0.9 限流约定
- 目标态限流计数走 Redis,超限返回 `429` 并带 `Retry-After` 响应头。
- 当前代码已先对 `POST /auth/send-code` 落地限流,规则和返回口径按下表执行;默认本地内存,配置 `APP_STATE_STORE_PROVIDER=redis` 后切 Redis,不改接口协议。
- 建议阈值:

| scene | 维度 | 规则 |
|---|---|---|
| `send_code` | `scene + account` | `1次/60s`, `5次/10m` |
| `send_code` | `deviceId` | `3次/10m` |
| `send_code` | `IP` | `10次/10m` |
| `order_create` | `userId` | `5次/1m` |
| `reservation_create` | `userId + shopId` | `3次/10m` |
| `device_register` | `userId + deviceUid` | `10次/1m` |
| `ws_connect` | `userId + IP` | `20次/1m` |
| `privacy_export` | `userId` | `3次/24h` |
| `privacy_delete` | `userId` | `1次/30d` |

- 当前已真正生效的只有 `send_code`;其他场景仍是目标态设计。
- 回调接口不做通用 `429`,而是靠验签、来源校验和幂等去重收口,别把 Stripe/微信的重试当攻击直接一刀切。

### 6.1 WebSocket 消息协议(`/ws/notify`)

握手约定:
- 浏览器先调 `POST /ws/ticket`,拿到 **60 秒有效、单次使用** 的 `ticket`。
- 再连接 `wss://{host}/api/c/v1/ws/notify?ticket=xxx`。
- 连接建立后服务端立即下发 `welcome`,并回补最近未读摘要。
- 心跳周期 `30s`,连续 `60s` 未收到 `ping/pong` 视为断连。

**服务端 `welcome`**:
```json
{
  "type": "welcome",
  "messageId": "ws_01J2M9WELCOME",
  "ts": 1752048000000,
  "data": {
    "connectionId": "conn_001",
    "userId": 10001,
    "heartbeatSeconds": 30
  }
}
```

**统一消息包**:
```json
{
  "type": "notification.new",
  "messageId": "ws_01J2M9ABCDEF",
  "ts": 1752048002000,
  "ackRequired": true,
  "data": {}
}
```

**客户端心跳 / ACK**:
```json
{ "type": "ping", "ts": 1752048010000 }
{ "type": "ack", "messageId": "ws_01J2M9ABCDEF" }
```

**服务端 `pong`**:
```json
{ "type": "pong", "ts": 1752048010001 }
```

消息类型:

| type | 说明 | `data` 核心字段 |
|---|---|---|
| `welcome` | 连接建立成功 | `connectionId`, `heartbeatSeconds` |
| `notification.new` | 新站内通知 | `notificationId`, `notificationType`, `title`, `content`, `refType`, `refId` |
| `message.new` | 私信新消息 | `conversationId`, `messageId`, `fromUserId`, `preview` |
| `reservation.reminder` | 预订提醒 | `reservationId`, `reserveTime`, `shopName` |
| `system.kickout` | 强制下线 / 多端互斥 | `reason`, `reasonCode` |
| `pong` | 心跳响应 | `ts` |

关闭与重连:
- `4001`: ticket 无效或已过期。
- `4003`: 未登录或用户状态异常。
- `4008`: 连接被后登录端顶掉。
- 前端收到非业务关闭码时采用指数退避重连,重连前重新申请 `ws/ticket`。

转发和取消转发都按当前 `X-Region` 锁定 `audit_status=1 AND status=1 AND is_deleted=FALSE` 的帖子；跨区域、待审、驳回、下架、已删除或不存在统一返回 `404`。每次关系变化后，`post.repost_count` 都从 `post_repost` 真实关系数重算；响应返回 `postId`、`reposted`、`repostCount`，帖子列表和详情同时返回 `repostCount` 与 `repostedByCurrentUser`。

话题请求统一读取 `X-Region`，只公开 `status=1 AND merged_to_id IS NULL` 的当前区域数据。非法排序、屏蔽名称发帖等参数错误返回 400；登录接口无会话返回 401；跨区域、屏蔽、已合并或不存在话题返回 404；管理端改名撞同区域唯一键返回 409。

热榜公式固定为 `post_count_7d * 20 + like_count_7d * 3 + comment_count_7d * 5 + (recommended ? 100 : 0)`。`pinnedSort > 0` 的话题优先，普通项按 `score DESC,follower_count DESC,id DESC`。快照落 `topic_hot_snapshot`，每小时按 CN/EU 独立增量重算；首次读取空区域同步兜底，事务替换失败保留旧快照，不依赖 Redis。

### 4.2 `POST /pay/notify/:channel` 统一归一化字段

> 第三方原始回调报文按各渠道协议提交,验签与解析由支付适配层完成。下面是**业务层归一化后的字段口径**,不是要求 Stripe/支付宝真的给你发成这个 JSON,别自己骗自己。

**统一字段**:
```json
{
  "channel": "stripe",
  "eventType": "payment.succeeded",
  "orderNo": "DP202607090001",
  "channelTxn": "pi_3Pxxx",
```

处理约束:

### 4.3 分渠道回调原始报文与验签摘要

#### 4.3.1 Stripe

**请求头**:
- `Stripe-Signature`

**原始 body 示例**:
```json
{
  "id": "evt_1Pxxx",
  "type": "payment_intent.succeeded",
  "created": 1752048062,
  "data": {
    "object": {
      "id": "pi_3Pxxx",
      "object": "payment_intent",
      "amount": 2990,
      "currency": "eur",
      "metadata": {
        "orderNo": "DP202607090001"
      }
    }
  }
}
```

验签摘要:
- 必须用**原始请求体** + endpoint secret 校验 `Stripe-Signature`,别先 JSON 解析再验签。
- 业务映射: `metadata.orderNo -> orderNo`,`data.object.id -> channelTxn`,`amount -> 分单位金额`。
- 成功类事件一期只处理 `payment_intent.succeeded`;退款类按 `charge.refunded` 或对账任务补。
- 返回任意 `2xx` 即视为已接收。

#### 4.3.2 PayPal

**请求头**:
- `PAYPAL-TRANSMISSION-ID`
- `PAYPAL-TRANSMISSION-TIME`
- `PAYPAL-TRANSMISSION-SIG`
- `PAYPAL-AUTH-ALGO`
- `PAYPAL-CERT-URL`

**原始 body 示例**:
```json
{
  "id": "WH-1AB23456CD7890123-0XY12345AB6789012",
  "event_type": "PAYMENT.CAPTURE.COMPLETED",
  "resource_type": "capture",
  "create_time": "2026-07-09T08:01:02Z",
  "resource": {
    "id": "4FJ12345AB6789012",
    "status": "COMPLETED",
    "amount": {
      "currency_code": "EUR",
      "value": "29.90"
    },
    "custom_id": "DP202607090001"
  }
}
```

验签摘要:
- 用上述请求头 + 原始 body + 本地 `webhook_id` 调 PayPal 的 `verify-webhook-signature` 验签接口。
- 业务映射: `resource.custom_id -> orderNo`,`resource.id -> channelTxn`,`resource.amount.value -> amount`。
- 一期主处理 `PAYMENT.CAPTURE.COMPLETED`;退款走 `PAYMENT.CAPTURE.REFUNDED`。
- 成功处理后返回 `200 OK`。

#### 4.3.3 支付宝

**Content-Type**:`application/x-www-form-urlencoded`

**原始 body 示例**:
```text
notify_time=2026-07-09 16:01:02&notify_type=trade_status_sync&notify_id=2026070900222abcdef&charset=UTF-8&version=1.0&app_id=2021000118630012&out_trade_no=DP202607090001&trade_no=2026070922001490010523001111&trade_status=TRADE_SUCCESS&total_amount=29.90&seller_id=2088102177846880&sign_type=RSA2&sign=Base64Signature
```

验签摘要:
- 对除 `sign` 外的通知参数做 RSA2 验签,并校验 `app_id` / `seller_id` / `out_trade_no` / `total_amount`。
- 业务映射: `out_trade_no -> orderNo`,`trade_no -> channelTxn`,`trade_status -> status`。
- 仅 `TRADE_SUCCESS`、`TRADE_FINISHED` 视为成功态。
- 处理成功必须返回纯文本 `success`。

#### 4.3.4 微信支付 v3

**请求头**:
- `Wechatpay-Signature`
- `Wechatpay-Timestamp`
- `Wechatpay-Nonce`
- `Wechatpay-Serial`

**原始 body 示例**:
```json
{
  "id": "EV-202607090001",
  "create_time": "2026-07-09T16:01:02+08:00",
  "event_type": "TRANSACTION.SUCCESS",
  "resource_type": "encrypt-resource",
  "summary": "支付成功",
  "resource": {
    "algorithm": "AEAD_AES_256_GCM",
    "ciphertext": "....",
    "associated_data": "transaction",
    "nonce": "nOnCe123"
  }
}
```

**解密后的业务报文**:
```json
{
  "mchid": "1900000109",
  "out_trade_no": "DP202607090001",
  "transaction_id": "4200002455202607091234567890",
  "trade_state": "SUCCESS",
  "success_time": "2026-07-09T16:01:02+08:00",
  "amount": {
    "total": 2990,
    "payer_total": 2990,
    "currency": "CNY"
  }
}
```

验签摘要:
- 先用平台证书验 `Wechatpay-*` 头签名,再用 APIv3 key 解密 `resource.ciphertext`。
- 业务映射: `out_trade_no -> orderNo`,`transaction_id -> channelTxn`,`amount.total -> 分单位金额`。
- 一期只把 `TRANSACTION.SUCCESS` + `trade_state=SUCCESS` 视为成功,其余走失败/补偿。
- 处理成功返回 `200` 或 `204`。

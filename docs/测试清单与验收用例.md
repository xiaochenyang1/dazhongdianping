## 1. 文档目标

这份文档就是把“要测什么、怎么判定过没过、上线前哪些必须绿”说清楚。

前面那堆设计文档解决的是“应该怎么做”，这份文档解决的是“你说你做完了，那拿什么证明”。不然最后就会变成开发说没问题、测试说像有问题、线上一跑全露馅。

## 2. 测试范围

一期建议把测试分三层:

- **M1/M2 必测**:浏览链路、认证鉴权、点评与审核、基础用户中心。
- **交易前置必测**:支付回调、预订状态机、WebSocket、隐私中心。哪怕这些不是最早上线,也要先把用例定好,免得后面边做边编。
- **横切能力必测**:区域隔离、幂等、限流、审计日志、告警、回滚。

## 3. 测试策略

### 3.1 优先级定义

| 优先级 | 含义 | 处理原则 |
|---|---|---|
| P0 | 核心链路,挂了不能上线 | 必须全通过 |
| P1 | 重要功能,挂了需要评估是否阻塞 | 原则上通过,必要时带风险上线审批 |
| P2 | 体验或边缘能力 | 可排到后续修复,但要登记缺陷 |

### 3.2 测试类型

| 类型 | 说明 |
|---|---|
| 功能测试 | 接口、页面、流程是否按预期运行 |
| 接口测试 | 参数校验、错误码、幂等、限流、鉴权 |
| 集成测试 | 第三方、对象存储、消息、回调、审核流 |
| 回归测试 | 关键功能改动后整链路重跑 |
| 冒烟测试 | 上线前后快速确认核心能力没死 |

### 3.3 上线闸门

下面这些没过,别上线:

- P0 用例全部通过
- 新增/变更接口有冒烟结果
- 关键回调、登录、审核、回滚路径已验证
- 告警、日志、回调地址、环境变量已核对

## 4. 测试环境矩阵

| 环境 | 用途 | 数据要求 |
|---|---|---|
| `dev` | 开发自测 | 可脏数据 |
| `test` | 联调与功能测试 | 需要基础分类/城市/种子商户 |
| `pre` | 上线前回归、演练 | 尽量贴近生产配置 |
| `prod` | 线上 | 只做冒烟,不做破坏性验证 |

补充要求:

- `test/pre` 都要有可用的对象存储、短信/邮件测试通道、后台审核账号。
- 支付、推送、回调类能力至少要有 **sandbox/mock** 或可演练替代链路。
- `CN/EU` 区域数据都要准备样例,别只测一边就装全量通过。

| TOPIC-006 | P0 | 首次读取与定时重算 | 无快照同步兜底；CN/EU 每小时独立增量重算，单区失败不阻断另一区 |
| TOPIC-007 | P0 | 快照事务替换失败 | 旧 `topic_hot_snapshot` 回滚保留，不出现空榜冒充成功 |
| TOPIC-008 | P0 | 管理端合并 | 帖子/关注关系去重迁移，目标计数按真实行数，源标记已合并；操作不可逆 |
| TOPIC-009 | P1 | 三端能力边界 | Flutter 可关注；PC Web 只读无写按钮；管理端可筛选、推荐、置顶、屏蔽、合并和重算 |
| TOPIC-010 | P1 | 非目标能力确认 | 无独立话题 Feed、无话题更新通知、热榜不依赖 Redis |

### 5.11 幂等、限流与审计

| ID | 优先级 | 场景 | 预期结果 |
|---|---|---|---|
| SAFE-001 | P0 | 同一 `Idempotency-Key` + 同请求体重复提交 | 返回首个结果(当前后端自动化已覆盖点评创建场景) |
| SAFE-002 | P0 | 同一 `Idempotency-Key` + 不同请求体 | 返回 `409`(当前后端自动化已覆盖点评创建场景) |
| SAFE-003 | P0 | 验证码/下单/预订超限 | 返回 `429` |
| SAFE-004 | P1 | 后台审核动作 | 审计日志完整记录操作人/动作/时间 |
| SAFE-005 | P1 | 回调异常 | 记录 traceId、原始报文、告警信息 |

## 6. 关键验收用例

### 6.1 用例 A:游客写点评被拦截后恢复

前置条件:

- 用户未登录
- 商户详情页可访问

步骤:

1. 游客进入商户详情页。
2. 点击“写点评”。
3. 系统弹出登录浮层。
4. 用户完成登录。
5. 系统自动回到写点评页。

期望:

- 不发生整页跳转丢上下文。
- 登录成功后可继续提交点评。
- 失败时有明确错误提示。

### 6.2 用例 B:点评提交到审核通过

前置条件:

- 用户已登录
- 对象存储可用
- 管理端审核账号可登录

步骤:

1. 用户提交一条带图点评。
2. 系统返回待审状态。
3. 管理端看到审核任务。
4. 审核员点击通过。
5. 前台刷新商户详情页。

期望:

- 点评状态流转正确。
- 商户评分和点评数聚合正确。
- 审核日志完整。

### 6.3 用例 C:支付回调重复通知

前置条件:

- 存在一笔待支付订单
- 支付渠道 sandbox/mock 已可触发回调

步骤:

1. 发送一次成功回调。
2. 再发送同样的回调报文。

期望:

- 第一次更新订单成功。
- 第二次命中幂等,不重复入账。
- 审计日志能看到回调处理结果。

### 6.4 用例 D:WebSocket 断线重连

前置条件:

- 用户已登录
- `ws/ticket` 接口可用

步骤:

1. 用户建连成功。
2. 手工断开连接。
3. 前端重新申请 ticket。
4. 前端重连成功。

期望:

- 新连接可正常收到 `welcome`。
- 未读消息可补拉或回补摘要。

### 6.5 用例 E:删除任务撤销

前置条件:

- 用户已登录
- `scene=delete` 验证码可用

步骤:

1. 用户发起删除任务。
2. 系统进入冷静期。
3. 用户在冷静期内点击撤销。

期望:

- 任务状态改为已取消。
- 账号仍可登录。
- 审计日志记录撤销动作。

## 7. 非功能测试要求

### 7.1 性能与稳定性

- 登录接口、商户列表、商户详情、点评列表做基本压测。
- WebSocket 至少验证单节点连接上限和心跳稳定性。
- 导出任务验证大数据量场景的超时与失败恢复。

### 7.2 安全

- token 过期和越权访问必须拦截。
- 文件上传必须校验类型、大小、内容。
- 回调必须验签、验金额、验订单归属。
- 后台和商户端关键操作必须落审计。

### 7.3 数据一致性

- 订单、预订、导出任务、删除任务、设备注册都要验证幂等。
- 商户评分聚合要验证并发更新。
- 区域数据不能串区,尤其是列表、榜单、热词。
- 门店草稿审核前线上 `shop/shop_photo/dish` 必须不变；通过时三张表同事务应用，任何一步失败都不能留下半套数据。
- 修改门店必须保留评分、点评数、团购聚合字段，并用 `base_updated_at` 阻止旧草稿覆盖新版本。

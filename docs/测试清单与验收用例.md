## 2. 测试范围

一期建议把测试分三层:

- **M1/M2 必测**:浏览链路、认证鉴权、点评与审核、基础用户中心。
- **交易前置必测**:支付回调、预订状态机、WebSocket、隐私中心。哪怕这些不是最早上线,也要先把用例定好,免得后面边做边编。
- **横切能力必测**:区域隔离、幂等、限流、审计日志、告警、回滚。

## 3. 测试策略

### 3.1 优先级定义

| 优先级 | 含义 | 处理原则 |
|---|---|---|
| P0 | 核心链路,挂了不能上线 | 必须全通过 |
| P1 | 重要功能,挂了需要评估是否阻塞 | 原则上通过,必要时带风险上线审批 |
| P2 | 体验或边缘能力 | 可排到后续修复,但要登记缺陷 |

### 3.2 测试类型

| 类型 | 说明 |
|---|---|
| 功能测试 | 接口、页面、流程是否按预期运行 |
| 接口测试 | 参数校验、错误码、幂等、限流、鉴权 |
| 集成测试 | 第三方、对象存储、消息、回调、审核流 |
| 回归测试 | 关键功能改动后整链路重跑 |
| 冒烟测试 | 上线前后快速确认核心能力没死 |

### 3.3 上线闸门

下面这些没过,别上线:

- P0 用例全部通过
- 新增/变更接口有冒烟结果
- 关键回调、登录、审核、回滚路径已验证
- 告警、日志、回调地址、环境变量已核对

## 4. 测试环境矩阵

| 环境 | 用途 | 数据要求 |
|---|---|---|
| `dev` | 开发自测 | 可脏数据 |
| `test` | 联调与功能测试 | 需要基础分类/城市/种子商户 |
| `pre` | 上线前回归、演练 | 尽量贴近生产配置 |
| `prod` | 线上 | 只做冒烟,不做破坏性验证 |

补充要求:

- `test/pre` 都要有可用的对象存储、短信/邮件测试通道、后台审核账号。
- 支付、推送、回调类能力至少要有 **sandbox/mock** 或可演练替代链路。
- `CN/EU` 区域数据都要准备样例,别只测一边就装全量通过。

| PRIV-006 | P0 | 注销到期清理话题关注 | 本人 `topic_follow` 清零，所有相关话题 `follower_count` 等于真实剩余行数，`post_topic/topic_hot_snapshot` 保留 |
| PRIV-007 | P0 | 冷静期内撤销 | 任务变已取消，账号保持可用（后端自动化 + Web E2E 已覆盖） |
| PRIV-008 | P0 | 冷静期到期执行删除 | 登录态失效、数据匿名化（后端自动化已覆盖） |

### 5.11 社区帖子转发

| ID | 优先级 | 场景 | 预期结果 |
|---|---|---|---|
| REPOST-001 | P0 | 游客、跨区用户或目标帖子非公开时请求转发 | 游客返回 `401`；跨区、待审、停用或删除帖子不可转发 |
| REPOST-002 | P0 | 同一用户重复 `POST /posts/:id/repost` | 请求幂等，只保留一条关系，返回 `reposted=true`，计数等于真实关系行数 |
| REPOST-003 | P0 | 同一用户重复 `DELETE /posts/:id/repost` | 请求幂等，关系删除后返回 `reposted=false`，计数不小于 0 且等于真实关系行数 |
| REPOST-004 | P1 | 登录用户读取帖子列表、圈子帖子和详情 | 返回真实 `repostCount` 与当前用户 `repostedByCurrentUser`；列表状态批量查询，不引入逐帖查询 |
| REPOST-005 | P1 | 转发或取消转发带话题的帖子 | 标记相关话题热榜需要刷新，后续快照反映真实互动变化 |
| REPOST-006 | P1 | Flutter 帖子详情转发与取消转发 | 按钮、计数和状态即时更新；页面销毁后的异步响应不触发无效 `setState` |

## 6. 关键验收用例

### 6.1 用例 A:游客写点评被拦截后恢复

前置条件:

- 用户未登录
- 商户详情页可访问

步骤:

1. 游客进入商户详情页。
2. 点击“写点评”。
3. 系统弹出登录浮层。
4. 用户完成登录。
5. 系统自动回到写点评页。

期望:

- 不发生整页跳转丢上下文。
- 登录成功后可继续提交点评。
- 失败时有明确错误提示。

### 6.2 用例 B:点评提交到审核通过

前置条件:

- 用户已登录
- 对象存储可用
- 管理端审核账号可登录

步骤:

1. 用户提交一条带图点评。
2. 系统返回待审状态。
3. 管理端看到审核任务。
4. 审核员点击通过。
5. 前台刷新商户详情页。

期望:

- 点评状态流转正确。
- 商户评分和点评数聚合正确。
- 审核日志完整。

### 6.3 用例 C:支付回调重复通知

前置条件:

- 存在一笔待支付订单
- 支付渠道 sandbox/mock 已可触发回调

步骤:

1. 发送一次成功回调。
2. 再发送同样的回调报文。

期望:

- 第一次更新订单成功。
- 第二次命中幂等,不重复入账。
- 审计日志能看到回调处理结果。

### 6.4 用例 D:WebSocket 断线重连

前置条件:

- 用户已登录
- `ws/ticket` 接口可用

步骤:

1. 用户建连成功。
2. 手工断开连接。
3. 前端重新申请 ticket。
4. 前端重连成功。

期望:

- 新连接可正常收到 `welcome`。
- 未读消息可补拉或回补摘要。

### 6.5 用例 E:删除任务撤销

前置条件:

- 用户已登录
- `scene=delete` 验证码可用

步骤:

1. 用户发起删除任务。
2. 系统进入冷静期。
3. 用户在冷静期内点击撤销。

期望:

- 任务状态改为已取消。
- 账号仍可登录。
- 审计日志记录撤销动作。

## 7. 非功能测试要求

### 7.1 性能与稳定性

- 登录接口、商户列表、商户详情、点评列表做基本压测。
- WebSocket 至少验证单节点连接上限和心跳稳定性。
- 导出任务验证大数据量场景的超时与失败恢复。

### 7.2 安全

- token 过期和越权访问必须拦截。
- 文件上传必须校验类型、大小、内容。
- 回调必须验签、验金额、验订单归属。
- 后台和商户端关键操作必须落审计。

### 7.3 数据一致性

- 订单、预订、导出任务、删除任务、设备注册都要验证幂等。
- 商户评分聚合要验证并发更新。
- 区域数据不能串区,尤其是列表、榜单、热词。
- 相似推荐不能混入自身、跨区、停用或引用停用基础数据的门店，排序与距离必须稳定可复验。
- 帖子 `repost_count` 必须以 `post_repost` 真实关系行数为准，重复转发/取消和用户注销都不能留下漂移计数。
- 门店草稿审核前线上 `shop/shop_photo/dish` 必须不变；通过时三张表同事务应用，任何一步失败都不能留下半套数据。
- 修改门店必须保留评分、点评数、团购聚合字段，并用 `base_updated_at` 阻止旧草稿覆盖新版本。

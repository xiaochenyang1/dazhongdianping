# 大众点评(仿)项目 · 权限矩阵

> 版本:v0.7(同步 Banner + 搜索热词管理闭环)
> 最后更新:2026-07-21
> 配套文档:`需求文档.md`、`接口设计.md`、`业务流程与状态机.md`、`术语与枚举规范.md`

## 1. 文档目标

把“谁能干什么”说清楚。这个不补,后面接口写着写着就会出现游客能删帖、店员能改资质、审核员能随手改权限这种离谱剧情。

## 2. 角色定义

| 角色 | 说明 | 登录前缀/入口 |
|---|---|---|
| 游客 | 未登录 C 端用户 | `/api/c/v1` |
| C 端登录用户 | 已登录普通用户 | `/api/c/v1` |
| 商户主账号 | 商户主体负责人,管理门店与员工 | `/api/b/v1` |
| 商户店员 | 门店核销、预订处理、点评回复 | `/api/b/v1` |
| 运营审核员 | 处理资质、团购、内容审核 | `/api/admin/v1` |
| 运营管理员 | 运营配置、数据管理、账号管理 | `/api/admin/v1` |
| 超级管理员 | 管理员与权限体系维护 | `/api/admin/v1` |
| 系统任务 | 定时任务、回调处理、同步程序 | 内部调用 |

## 3. 区域权限规则

- 游客和 C 端用户默认跟随 `request_region` 访问公开数据。
- C 端登录用户可以跨区域拥有行为数据,但单次请求只在一个区域上下文内执行。
- `preferred_region` 只是用户偏好,不扩大访问权限。
- 商户主账号一期绑定一个经营区域;当前 B 端工作台请求要求 `X-Region` 与该经营区域一致,错区请求直接拒绝;如后续支持跨区,需要门店级授权。
- 管理端区域范围已通过 `admin_region_scope` 落地；区域型接口会校验 `X-Region`，EU-only 管理员访问 CN 数据返回 `403`。城市/门店级细粒度数据权限仍未实现。

## 5. 商户端角色模型(一期口径)

当前按四种商户侧角色落地:

| 角色 | 核心能力 | 门店范围 |
|---|---|---|
| 主账号 | 主体资料、门店管理、员工管理、团购管理、财务查看 | 全部门店 |
| 店长 | 门店编辑、团购管理、预订处理、点评回复 | 全部或指定门店 |
| 核销员 | 券码核销、预订到店确认、爽约标记 | 指定门店 |
| 客服/运营 | 点评回复、申诉、消息处理 | 全部或指定门店 |

对应当前实际库表:

- `merchant_operator`：商户主账号与员工操作者。
- `merchant_role`：商户角色，权限集合保存在当前 `permissions` 字段。
- `merchant_operator_role`：操作者与角色关联。
- `merchant_operator_shop`：操作者的全部/指定门店范围关联。

## 6. 管理端数据库 RBAC(当前已落地)

### 6.1 当前内置角色与权限模型

- `super_admin`（超级管理员）：当前全部权限点，可维护管理员、角色和权限分配。
- `content_auditor`（内容审核员）：点评、帖子、本地达人认证和商户点评申诉审核权限。
- `merchant_auditor`（商户审核员）：商户资质、团购和门店变更审核权限。
- `operations_manager`（运营管理员）：Banner、搜索热词、榜单、成长、官方圈子和话题治理权限。
- `data_operator`（数据管理员）：门店、导入批次、分类/城市/商圈、订单退款查询/仲裁/对账补偿和搜索索引权限。

权限通过 `admin_user → admin_user_role → admin_role → admin_role_permission → admin_permission` 计算，区域范围从 `admin_region_scope` 读取。`GET /api/admin/v1/auth/me` 返回实时身份、权限和区域；`GET /api/admin/v1/menus`、前端路由和后端 API 使用同一权限码。角色停用后旧 token 仍能请求 `auth/me` 并得到 `200`，但权限会重新计算并收回，固定受限 API 返回 `403`，动态审核列表可返回 `200` 空结果；管理员账号停用后，旧 token 才会在下一次请求返回 `401`，前端清理 `localStorage` 并回到登录页。

### 6.2 当前权限点分类

- 菜单权限:能看到哪个后台菜单，由 `/menus` 过滤无权限叶子和空分组。
- 动作/API 权限:例如 `audit:review:write`、`audit:expert_certification:write`、`data:shop:import`、`data:geo:read`、`data:geo:write`、`system:admin:write`，由后端接口校验。
- 区域数据权限:当前已实现 `CN/EU` 范围；城市、门店等更细粒度范围仍待实现。

`data:geo:read` 控制基础数据菜单、`/data/meta` 路由和三类列表接口；`data:geo:write` 控制新建、编辑、启停与删除。只有写权限而没有读权限不会获得页面入口；所有 API 仍独立校验写权限和 `X-Region` 范围。
`data:order:read` 控制管理端 `/data/orders` 菜单、路由和 `GET /api/admin/v1/orders` 查询接口，返回当前区域的订单、支付流水和退款状态。
`data:order:write` 控制平台退款仲裁 `POST /api/admin/v1/orders/{id}/refund-audit` 与对账补偿 `POST /api/admin/v1/orders/reconcile`；通过退款会同步订单/券/库存并写 `refund_approve` 审计日志，驳回写 `refund_reject`；对账补偿关闭超时未支付订单、恢复库存、标记失败支付流水并写 `trade_reconcile` 审计日志。定时任务每分钟也会自动执行同一补偿逻辑。
`operations:banner:read/write` 控制管理端 `/operations/banners` 菜单、路由和 `GET/POST/PUT/DELETE /api/admin/v1/banners*`，统一按 `X-Region` 隔离；`cityId=null` 表示区域通用 Banner。
`operations:hotword:read/write` 控制管理端 `/operations/hotwords` 菜单、路由和 `GET/POST/PUT/DELETE /api/admin/v1/search/hotwords*`；公开端 `/api/c/v1/search/hot` 会优先返回当前区域启用热词，删空后再回退统计结果。
`audit:expert_certification:read/write` 控制管理端 `/audit/expert-certifications` 菜单、路由和统一审核接口中的 `biz_type=7` 达人认证任务。
`system:audit_log:read` 控制管理端 `/system/audit-logs` 菜单、路由和 `GET /api/admin/v1/audit/logs` 查询接口。
`system:privacy_task:read` 控制管理端 `/system/privacy-tasks` 菜单、路由和 `GET /api/admin/v1/privacy/tasks` 查询接口。
`system:user:read` 控制管理端 `/system/users` 菜单、路由和 `GET /api/admin/v1/users`、`GET /api/admin/v1/users/{userId}` 查询接口；`system:user:write` 控制 `PUT /api/admin/v1/users/{userId}/status` 封禁/解封动作。封禁必须填写原因并写入审计日志(`user_ban`/`user_unban`)，同时吊销该用户全部会话；已注销(匿名化)用户不可变更状态。
`audit:user_appeal:read/write` 控制管理端 `/audit/user-appeals` 菜单、路由和统一审核接口中的 `biz_type=8` 用户封禁申诉任务。通过申诉会立即解封该用户并写入审计日志(`audit_user_appeal_pass`)；驳回会记录原因(`audit_user_appeal_reject`)并回显给用户。管理员直接解封时，待审申诉自动置为已通过、对应审核任务失效。申诉通过/驳回/直接解封都会给用户写 `account.ban_appeal` 类型站内通知（走通知模块，含 WebSocket 推送），用户恢复登录后可在通知列表看到结果；申诉提交/查询响应会带出最近一次 `user_ban` 审计日志中的封禁原因。

## 7. 接口鉴权建议

- C 端统一 JWT,游客态可不带 token。
- B 端与管理端建议独立 token 空间,别混用。
- WebSocket 建立连接前必须完成 token 鉴权。
- 第三方支付回调、导入回调等非人类请求要用签名或内部鉴权,不是拿 JWT 糊弄。

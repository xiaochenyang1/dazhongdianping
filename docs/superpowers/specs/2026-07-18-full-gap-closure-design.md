# 全量功能缺口收口设计

> **状态：** 总体方向已由用户选择方案 A；本文用于确认实施边界、子项目顺序和最终验收口径。

## 目标

以 `docs/需求文档.md`、接口/数据库/权限/流程/第三方和上线文档为完整需求来源，对当前代码进行逐项核验，修正文档中的过期状态，并补齐所有尚未实现的仓库内功能。真实云资源、支付、地图、推送、短信邮件、内容审核和部署能力必须在获得有效账号与凭证后完成真实环境验收，不能用 mock、配置占位或“代码入口已留”冒充完成。

## 当前基线

当前仓库已经具备 M1-M7 的主要本地闭环：公开浏览、搜索、认证、点评、成长体系、榜单、收藏、团购、预订、商户经营、WebSocket 通知、隐私中心、帖子、关注关系、私信、官方圈子和话题热榜。

基线命令 `scripts/ci/verify-all.ps1 -IncludeFlutter` 已通过，覆盖后端、PC Web、管理端、商户端和 Flutter 的测试、静态检查与构建。该结果只能证明现有自动化覆盖范围通过，不能证明未编写的功能或目标环境联调已经完成。

已确认的主要问题：

- 文档状态落后，同一功能在不同文档甚至同一文档中同时被标为已完成和未完成。
- PC Web 部分高级筛选、分页、点评排序、分享、相似推荐和 SEO 预渲染未完成。
- 商户后端已具备入驻与员工权限，但商户前端缺少入驻、资质和员工管理闭环。
- 管理端仍使用配置账号和内存会话，缺少管理员 RBAC、用户/订单/基础数据/运营活动管理及审计查询。
- 社交侧缺少帖子转发、评论盖楼、@ 提醒、完整通知聚合和达人认证。
- Flutter 国际化覆盖不完整，真实支付、Google Maps、FCM/APNs 尚未接入。
- 邮件、短信、内容审核、CDN、真实支付及部署仍缺少供应商实现或真实环境验证。

## 总体方案

采用“证据矩阵 + 独立子项目 + 依赖顺序验收”的方式收口，不把全部缺口塞进一个超大改动。

每项需求必须记录四类证据：

1. **实现证据：** 对应控制器、服务、页面、数据库结构和配置。
2. **自动化证据：** 能证明核心规则、权限、状态机和异常路径的测试。
3. **运行证据：** 构建结果、浏览器链路、基础设施 smoke 或真实供应商回调。
4. **文档证据：** 需求、接口、数据库、流程和当前完成状态已同步一致。

只有四类证据中与该需求相关的部分全部成立，状态才能改为“已完成”。外部能力如果只有适配器和契约测试，应明确标为“代码完成、真实联调未完成”。

## 子项目划分与顺序

### 1. 文档事实基线与验收矩阵

先修正 `需求文档.md`、`当前已完成功能与SQL导入说明.md`、根 `README.md` 和测试清单中的冲突。建立功能矩阵，区分：已完成、部分完成、仓库内未完成、依赖外部资源、明确不在一期范围。

该阶段不通过修改文档缩小原需求，只负责让文档准确描述代码和真实环境。

### 2. 商户端与管理端完整闭环

商户端补齐注册、资质提交与重提、审核状态、员工列表/创建/编辑/启停、角色和门店范围配置。

管理端补齐商户入驻审核页面、管理员数据库账号与 RBAC、分类/城市/商圈管理、用户治理、订单退款与对账查询、首页运营位、搜索热词、运营活动资源、审计日志和隐私任务审计。

权限校验必须同时在后端和前端导航层生效；前端隐藏按钮不能代替后端鉴权。

### 3. PC Web 产品缺口

补齐商户列表价格、评分、有团购、营业中等筛选及真实分页；距离排序必须使用真实坐标，不允许退化成商圈排序却仍显示“距离优先”。

商户详情补齐分享、附近相似推荐、点评最新/最热/评分/带图筛选和相册完整展示。社区公开页增加可抓取的预渲染产物、语义化元数据、canonical 和结构化数据。

### 4. 社区与消息闭环

帖子转发使用独立关系记录，支持原创与转发链路、计数幂等和隐私导出。评论增加父评论和回复对象，限制盖楼深度，删除时保持回复上下文可解释。

发帖和评论解析合法 `@用户`，生成去重通知。消息中心统一聚合点赞、评论、关注、@、系统、预订和私信事件，并保留 WebSocket 在线推送与 REST 离线补偿。

达人认证由管理端审核，公开用户资料和内容作者信息仅展示已通过且有效的认证。

### 5. Flutter 国际化、地图、支付与推送

将散落硬编码文案迁移到完整的简体、繁体和英文资源；日期、金额、电话号码和地址按区域格式化。翻译点评必须保存原文、目标语言和翻译来源，失败时回退原文。

Google Maps 仅在 EU APP 启用，支持门店标记、附近搜索和外部导航。无授权或无密钥时保留列表能力并返回明确原因。

支付由后端创建支付会话，客户端只负责展示供应商 SDK 页面；最终支付结果必须来自服务端验签回调。EU 首批实现 Stripe 和 PayPal，CN 实现支付宝和微信支付，区域本地方式通过同一适配层扩展。

Push 使用统一 `PushProvider`，Android 走 FCM、iOS 走 APNs。发送失败按可重试错误退避重试，确定失效 token 自动停用，站内通知始终保留为补偿来源。

### 6. 供应商与上线环境

邮件、短信、对象存储、CDN、内容审核、支付和推送都通过适配器隔离。供应商返回值统一包含 provider、requestId、原始错误码、是否可重试和审计摘要。

目标环境验收包括 MySQL、Redis、S3/OSS、CDN、域名证书、邮件短信、内容审核、支付回调、Push、SSH 发布和回滚。密钥只能来自环境变量或密钥管理服务，测试输出不得打印凭证。

预算、联系人、域名归属和供应商账号属于用户或组织必须提供的真实执行信息。缺少这些信息时，仓库开发可以继续，但对应上线项不得标记完成。

## 数据和接口原则

- 所有新增业务表同时维护 H2 测试结构和 MySQL 正式结构。
- 写操作必须考虑幂等、区域隔离、权限、审计、隐私导出和账号注销。
- 计数类字段由真实关系表重算或事务增减，不能只相信客户端传值。
- 外部回调必须使用原始请求体验签，保存可审计摘要并支持重复通知。
- 新接口沿用 `/api/c/v1`、`/api/b/v1`、`/api/admin/v1` 边界。
- 兼容现有客户端时优先增加可选字段；破坏性变更必须同步升级所有调用端。

## 错误处理

- 参数错误、权限不足、区域不符、状态冲突和外部服务故障使用稳定错误码区分。
- 外部服务可重试失败进入有限次数退避队列；不可重试失败直接记录并提供人工补偿入口。
- 支付、退款、审核和隐私任务失败不能留下半完成状态，必须通过事务或补偿恢复一致性。
- 页面只显示可操作的错误原因，不暴露密钥、原始签名、数据库信息或完整第三方报文。

## 测试策略

每个子项目采用测试先行：

- 后端覆盖服务规则、数据库集成、权限、区域隔离、幂等和状态机。
- Vue/Flutter 覆盖页面状态、表单校验、错误反馈和关键交互。
- 浏览器 E2E 覆盖用户、商户和管理员跨端主链路。
- 第三方适配器覆盖官方 sandbox 或本地签名契约，真实凭证环境再跑独立 smoke。
- 数据库变更同时执行 H2 全测和 MySQL 8 导库/启动 smoke。
- 每个子项目结束运行受影响测试；所有子项目完成后运行仓库全量验证和上线清单。

## 完成定义

最终完成必须同时满足：

1. 需求文档中所有一期功能均有实现和验收证据，或已由用户明确移出范围。
2. 文档之间不存在同一功能状态冲突。
3. 后端、三个 Web 端和 Flutter 全量测试、静态检查与构建通过。
4. 浏览器 E2E 覆盖用户、商户和管理端关键业务链路。
5. MySQL、Redis、对象存储、Elasticsearch 和发布回滚 smoke 通过。
6. 真实支付、地图、短信邮件、内容审核和 Push 在目标环境使用真实凭证通过。
7. 上线准备清单、联系人台账和预算台账已由有权限的负责人填写并核验。

在第 6、7 项没有真实证据前，只能报告仓库实现进度，不能宣称整个项目完成上线。

## 非目标

以下内容不因本轮收口自动扩展：点评视频、群聊、红包、语音通话、PC 地图、会员权益、签到、积分商城、多供应商实时双活。若后续要做，必须作为新需求单独设计。

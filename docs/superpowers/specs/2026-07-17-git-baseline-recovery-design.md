# Git 基线恢复设计

## 背景

当前工作区 `E:\tuowei\python\dazhongdianping` 存在 `.git` 目录，但该目录为空，不包含 `HEAD`、`config`、`objects`、`refs` 或 `logs`。项目没有远程仓库，也没有可用历史备份，因此旧提交历史无法恢复。

用户已确认接受当前通过全量验证的工作区作为新的 Git 历史起点。本设计只处理本地 Git 基线；测试环境发布、远程仓库创建和真实推送分别进入后续独立设计，不在本次范围内。

## 目标

- 在不伪造旧历史的前提下建立新的 `main` 分支。
- 保留空 `.git` 的检查证据，不直接删除。
- 确保首次提交不包含密钥、本地数据、依赖目录或构建产物。
- 将当前可运行、可测试的项目状态固化为可审计基线。
- 创建明确的首次提交与本地基线标签。

## 非目标

- 不恢复不存在的旧提交、作者或提交时间。
- 不创建或绑定 GitHub、GitLab 等远程仓库。
- 不推送代码，不创建 Pull Request。
- 不配置测试、预发或生产服务器。
- 不修改业务代码来迎合基线提交。
- 不在缺少真实身份时伪造 Git 作者信息。

## 方案选择

采用“经过审计的全新基线”方案：先检查工作区和忽略规则，保留空 `.git`，再初始化仓库、审计暂存内容、执行验证并提交。

未采用以下方案：

- 直接 `git init && git add .`：速度快，但可能误提交 `.env`、本地存储、依赖目录、构建产物或密钥。
- 只创建压缩包：不能提供增量记录、分支、标签和回滚能力。

## 执行流程

### 1. 前置状态确认

- 再次确认 `.git` 仍为空，且没有在设计期间出现新的 Git 元数据。
- 记录工作区根路径、当前日期和 `.git` 文件数量。
- 检查全局 Git 作者配置：`user.name` 与 `user.email`。
- 如果作者配置缺失，停止并向用户询问，不使用临时或虚构身份。

### 2. 忽略规则与敏感文件审计

审查现有 `.gitignore`，至少确保以下内容不会进入首次提交：

- `.env`、`.env.*` 中的本地或真实环境配置，允许显式的示例文件除外。
- 私钥、证书、keystore、服务账号 JSON 和 SSH 凭证。
- `node_modules`、`target`、`build`、`dist`、Flutter 临时目录和 IDE 缓存。
- `local-storage`、运行日志、上传文件、导出文件和测试运行产生的数据。
- 数据库本地文件、临时压缩包和发布产物。

审计分两层执行：

1. 按文件名和扩展名查找常见敏感文件。
2. 对候选源码执行敏感内容模式检查，包括私钥头、访问令牌、密码和云服务密钥格式。

发现疑似真实凭证时停止流程；先确认其性质并从候选提交中排除，不能为了让检查通过而静默删除用户文件。

### 3. 保留空 `.git`

- 确认 `.git` 的绝对路径位于工作区根目录。
- 确认目录中没有任何文件或子目录。
- 留档目录名使用执行时本地时间，格式固定为 `dazhongdianping-empty-git-yyyyMMdd-HHmmss`。例如执行时间为 2026-07-17 20:48:00 时，路径为：

```text
E:\tuowei\python\dazhongdianping-empty-git-20260717-204800
```

- 移动后再次确认工作区内 `.git` 不存在，留档路径存在且仍为空。

不得对非空 `.git` 执行该步骤；如果状态发生变化，必须停止并重新评估。

### 4. 初始化仓库

在工作区根目录执行：

```powershell
git init -b main
```

初始化后确认：

- 当前分支为 `main`。
- 仓库根目录与工作区根目录一致。
- 没有配置虚构远程地址。

### 5. 暂存审计

- 使用 `git status --short --untracked-files=all` 查看候选文件。
- 先执行暂存预览或分批暂存，再检查 `git diff --cached --stat` 与 `git diff --cached --name-status`。
- 重点确认 `.env`、密钥、本地数据、依赖目录和构建产物没有进入索引。
- 设计文档和实施计划应包含在首次基线提交中。

如果候选清单异常巨大或出现明显生成文件，先修正 `.gitignore`，清空暂存区后重新审计，不带病提交。

### 6. 基线验证

首次提交前执行：

```powershell
.\scripts\ci\verify-all.ps1 -IncludeFlutter
```

只有退出码为 `0` 且最终输出包含 `all requested checks passed` 才允许继续。失败时保留仓库和工作区，修复失败原因后重新运行；不能创建“先提交再说”的基线。

### 7. 创建首次提交与标签

首次提交信息固定为：

```text
chore: establish verified project baseline
```

提交后创建带说明的本地标签：

```text
baseline-2026-07-17
```

标签说明必须明确：这是旧历史不可恢复后的新起点，不代表项目最初创建日期。

### 8. 提交后验证

- 确认 `git log -1` 的提交信息、作者和时间正确。
- 确认标签指向首次提交。
- 运行 `git status --short`，要求没有非预期变更。
- 再次执行完整验证命令，确认初始化和提交没有改变项目行为。
- 如果验证产生被忽略的构建文件，只需确认其忽略状态；不得为追求绝对空目录而提交产物。

## 失败处理

- `.git` 不再为空：停止移动和初始化，先检查新增元数据来源。
- Git 作者配置缺失：停止提交并请求用户提供姓名和邮箱。
- 发现真实凭证：停止暂存和提交，报告文件路径与风险，不在日志中输出凭证内容。
- 全量验证失败：保留当前状态，按失败项修复后重新验证。
- 首次提交失败：不强行修改历史；排查钩子、身份或文件锁问题后重试。
- 标签已存在：检查其目标；不覆盖未知标签。

## 验收标准

1. 工作区拥有有效 Git 仓库，默认分支为 `main`。
2. 空 `.git` 已移动到工作区外留档，没有被静默删除。
3. 首次提交不包含凭证、本地运行数据、依赖目录或构建产物。
4. 首次提交信息为 `chore: establish verified project baseline`。
5. 本地标签 `baseline-2026-07-17` 指向首次提交，并说明旧历史不存在。
6. 提交前后完整验证均退出 `0`，输出 `all requested checks passed`。
7. 提交完成后工作区没有非预期变更。
8. 没有创建、绑定或推送任何远程仓库。

## 后续阶段

基线恢复完成后，单独设计并实施测试环境交付闭环：远程仓库、CI 触发、测试服务器凭证、版本化发布、健康检查和回滚演练。该阶段不得与本次 Git 初始化混合执行。

# M7 话题广场、热榜与话题关注设计

## 目标

在现有 `topic/post_topic` 基础上实现区域化话题广场、最近 7 天热榜、话题详情、话题关注和管理端治理。Flutter 提供完整发现与关注能力，PC Web 提供只读展示；不新增话题关注 Feed 或话题更新通知。

## 产品边界

- 用户发布帖子时可输入新话题，后端按当前区域自动建档。
- 话题严格绑定 `CN` 或 `EU`，不允许跨区域读取、关注或合并。
- 一期支持推荐、热榜、详情、帖子列表、关注/取消关注和“我关注的话题”。
- 不做话题关注动态流、话题更新通知、话题聊天、用户编辑话题资料或话题管理员。
- PC Web 只读，不提供关注或发帖操作。
- 热榜使用数据库快照，不依赖未落地的真实 Redis 环境。

## 架构

新增独立 `topic` 领域服务负责公开查询、关注、治理和热榜快照。帖子仍由现有 `community` 模块维护，发帖时通过 `post_topic` 关联话题。审核、删除、点赞和评论状态变化后触发相关话题热度刷新标记；定时任务按区域批量重算快照，读取无快照时使用 SQL 聚合兜底。

## 数据模型

### 扩展 `topic`

- `follower_count`：关注人数冗余计数。
- `recommended`：是否运营推荐。
- `pinned_sort`：大于 0 时优先于普通热榜，数值越大越靠前。
- `merged_to_id`：合并后的目标话题 ID，未合并为空。
- `status`：`1` 正常、`2` 屏蔽。
- `post_count`：审核通过、公开且未删除的关联帖子数。

区域内名称保持唯一。公开查询只返回 `status=1 AND merged_to_id IS NULL` 的话题。

### `topic_follow`

- `id`
- `topic_id`
- `user_id`
- `created_at`

唯一约束：`topic_id + user_id`。增加用户维度索引支持“我关注的话题”。

### `topic_hot_snapshot`

- `id`
- `topic_id`
- `region`
- `score`
- `post_count_7d`
- `like_count_7d`
- `comment_count_7d`
- `calculated_at`

唯一约束：`topic_id`。区域和分数建立排序索引。

## 热度规则

只统计最近 7 天内审核通过、状态正常、未删除的帖子：

```text
score = post_count_7d * 20
      + like_count_7d * 3
      + comment_count_7d * 5
      + (recommended ? 100 : 0)
```

- `pinned_sort > 0` 的话题排在普通话题之前，按 `pinned_sort DESC` 排序。
- 普通话题按 `score DESC, follower_count DESC, id DESC` 排序。
- 被屏蔽或已合并的话题不生成有效榜单项。
- 热榜定时任务默认每小时重算一次；应用启动后首次读取若无快照，执行当前区域同步兜底计算。
- 快照是派生数据，可安全重建，不参与隐私导出。

## C 端 API

- `GET /api/c/v1/topics`
  - 参数：`sort=recommended|hot|latest`、`page`、`pageSize`。
- `GET /api/c/v1/topics/hot`
  - 最近 7 天热榜，返回热度构成。
- `GET /api/c/v1/topics/following`
  - 登录用户关注的话题列表。
- `GET /api/c/v1/topics/{id}`
  - 话题详情。
- `GET /api/c/v1/topics/{id}/posts`
  - 话题公开帖子分页。
- `PUT /api/c/v1/topics/{id}/follow`
  - 关注，幂等。
- `DELETE /api/c/v1/topics/{id}/follow`
  - 取消关注，幂等。

话题响应包含：

```text
id/region/name/postCount/followerCount/recommended/pinnedSort/
followedByCurrentUser/hotScore/postCount7d/likeCount7d/commentCount7d
```

游客读取时 `followedByCurrentUser=false`。`following` 和写操作必须登录。

## 自动创建与帖子关联

- 发帖请求继续接受最多 5 个话题名称。
- 名称去除首尾空格并在同一请求内去重。
- 区域内按标准化后的名称精确匹配；不存在时自动创建正常话题。
- 已屏蔽名称不能被自动重新创建或关联，返回 400 并提示话题不可用。
- 已合并话题名称被输入时，自动关联到最终目标话题。
- 帖子编辑重新提交时先删除旧关联，再建立新关联。
- 帖子审核、重新提交、删除或下架后刷新相关话题 `post_count` 并标记快照待重算。

## 关注关系

- 首次关注插入 `topic_follow` 并增加 `follower_count`。
- 重复关注不增加计数。
- 实际取消关注后扣减计数，最低为 0。
- 屏蔽或合并后的源话题不能新增关注。
- 合并话题时源关注关系迁移到目标话题，利用唯一键去重后重新计算目标 `follower_count`。

## 管理端 API

- `GET /api/admin/v1/topics`
- `PUT /api/admin/v1/topics/{id}`：改名。
- `PUT /api/admin/v1/topics/{id}/recommendation`
- `PUT /api/admin/v1/topics/{id}/status`
- `POST /api/admin/v1/topics/{id}/merge`
- `POST /api/admin/v1/topics/recalculate-hot`

管理端所有操作受当前 `X-Region` 限制。

### 合并事务

1. 锁定源话题和目标话题。
2. 校验同区域、非自身、均未被合并。
3. 将源 `post_topic` 迁移至目标，冲突关联去重。
4. 将源 `topic_follow` 迁移至目标，冲突关注去重。
5. 设置源 `merged_to_id=目标 ID`、`status=2`。
6. 重新计算目标 `post_count/follower_count`。
7. 删除源话题热榜快照并重算目标快照。

任何一步失败则整体回滚。

## Flutter

- 社区页新增“话题”入口。
- 话题广场包含“推荐”“热榜”“已关注”三个 Tab。
- 游客打开“已关注”只显示登录引导，不调用受保护接口。
- 热榜卡片显示排名、热度分、7 天帖子/点赞/评论构成。
- 话题详情展示关注人数、热度、关注按钮和公开帖子。
- 关注/取消关注使用乐观状态，失败恢复原计数和状态。
- 帖子编辑器继续允许输入新话题，不开放独立创建话题表单。

## PC Web

- 新增 `/topics` 话题广场与热榜。
- 新增 `/topics/:id` 详情和公开帖子列表。
- 页面支持 SEO 标题与描述。
- 不展示关注、取消关注、创建或发帖控件。

## 管理端页面

- 按区域、状态、推荐状态和关键词筛选。
- 展示帖子数、关注数、热度和最近计算时间。
- 支持改名、推荐/取消推荐、置顶排序、屏蔽、合并和手动重算。
- 合并操作必须二次确认并明确展示源话题、目标话题及不可逆影响。

## 隐私与注销

- 新增 `topics` 隐私导出模块，包含本人关注的话题与关注时间。
- 账号注销时删除本人 `topic_follow`，按涉及话题重新计算 `follower_count`。
- 帖子与话题关联继续保留，帖子作者按现有注销规则匿名化。
- 热榜快照不包含用户身份，不需要逐用户清理。

## 错误处理

- 话题不存在、屏蔽、已合并或跨区域：公开读取和关注返回 404。
- 输入屏蔽话题名称：400。
- 重复关注/取消关注：200，返回真实状态和计数。
- 跨区域合并、合并自身或合并到已合并话题：400。
- 改名与区域内现有话题冲突：409。
- 快照重算失败保留旧快照并记录错误，不让公开榜单直接消失。

## 验收标准

1. 用户发帖可自动创建并关联区域话题。
2. 热榜按最近 7 天固定公式稳定排序，支持推荐和置顶。
3. 用户可关注/取消关注并查看“我关注的话题”。
4. 管理端可治理、合并并手动重算话题。
5. Flutter 完整互动，PC Web 保持只读。
6. CN/EU 数据不串区。
7. 隐私导出和账号注销正确治理话题关注。
8. 全量 CI、Flutter analyze 和构建全部通过。

# M5a 商户身份前端闭环设计

## 目标

复用已经完成的 M5a 后端接口，在 `merchant-web` 补齐商户注册、资质提交/重提、审核状态和员工管理，在 `admin-web` 补齐商户资质审核页面，使商户从注册到进入经营工作台形成真实跨端闭环。

## 范围

### merchant-web

- `/register`：商户主账号注册，支持 CN/EU、企业名称、联系人和联系电话。
- `/settlement`：登录后查看资质状态；未提交或驳回时可填写营业执照 URL、法人和最多 12 张门店照片 URL；待审时只读；通过后提供进入工作台入口。
- `/staffs`：仅拥有 `staff:manage` 权限的主账号可访问，支持员工分页、创建、编辑、启停、角色和门店范围配置。
- 登录页增加注册入口；待审商户登录后跳到 `/settlement`，而不是进入后再被一堆 `401` 糊脸。
- 商户布局从 `account/me` 获取权限，只显示当前操作者可用菜单。

### admin-web

- `/audit/merchant-applications`：按区域和状态分页查看申请。
- 展示公司、账号、法人、执照、门店照片、提交/审核时间和驳回原因。
- 待审申请支持通过或填写原因驳回；已经处理的申请只读。
- 菜单由后端 `GET /menus` 增加对应入口，避免路由存在但导航失踪。

## 交互与视觉

沿用现有橙色品牌、浅暖灰背景和卡片体系，采用“经营档案/审核工作台”的实用型视觉，不引入新组件库。

- 表单按身份信息、资质材料、权限范围分组，长表单不堆成一根水泥柱。
- 状态使用清晰的待提交、待审核、已通过、已驳回标签；颜色不是唯一信息来源，必须同时有文本。
- 动态数量使用等宽数字；标题启用平衡换行；按钮点击使用 `scale(0.96)`，只过渡 `transform/opacity/background-color`。
- 所有按钮和小控件至少 40px 命中区域；图片带低透明黑色描边。
- 弹层仅用于员工编辑和审核决策，关闭后恢复焦点；错误保留在当前表单附近。

## 数据流

1. 注册成功返回 access token，写入现有商户会话并携带注册区域跳转 `/settlement`。
2. `/settlement` 调 `GET /settle/status`。提交或重提后刷新状态，不人工伪造状态。
3. 审核管理员调用 `POST /merchant-applications/{merchantId}/audit`；后端事务同步申请和商户审核状态。
4. 已通过商户进入布局后加载 `account/me`、`roles` 和 `shops`。员工页面以这些结果生成角色和门店选项。
5. 员工创建/编辑后重新拉取列表；启停动作成功后只替换对应行。

## 类型与边界

- 为商户注册、资质、账号、角色、员工和门店建立显式 TypeScript 类型，禁止继续用 `Record<string, unknown>` 糊住核心身份数据。
- 注册密码前端最少 8 位，后端仍是最终权威。
- 指定门店范围 `shopScopeType=2` 时至少选择一家当前商户门店；全部门店为 `1`。
- 员工不能分配 `owner` 角色；前端过滤，后端继续二次拦截。
- 管理端驳回必须有非空原因。
- 区域来自当前会话/管理端区域上下文，跨区域申请不会出现在列表。

## 错误处理

- `401` 且商户资质未通过：清晰引导到资质页，不清空仍有效的注册会话。
- 会话真正失效：清空会话并回到登录页。
- 重复账号、非法门店范围和重复审核直接显示后端稳定消息。
- 提交期间禁用按钮，防止重复点击；网络失败保留用户已填内容。
- 员工编辑失败不关闭弹层；审核失败不提前改变卡片状态。

## 测试

- merchant-web 服务测试覆盖注册、资质、员工 API 请求结构。
- 页面测试覆盖注册成功跳转、驳回重提、待审只读、员工指定门店校验和启停。
- 路由测试覆盖游客、待审商户、无 `staff:manage` 员工和主账号。
- admin-web 页面测试覆盖按状态加载、通过、驳回原因校验和后端失败回显。
- 两个前端执行 `npm test` 与 `npm run build`；后端现有 M5a 测试继续作为接口权威。

## 非目标

本子项目不实现管理员数据库 RBAC、基础数据/用户/订单/运营活动管理，也不修改 M5b 经营业务规则。这些属于后续独立子项目。

## 完成判定

商户能够在空会话下注册、提交资质，管理员能够审核，审核通过后商户可进入工作台并维护员工；驳回可重提，权限和门店范围错误由前后端共同阻止；相关测试和构建全部通过。

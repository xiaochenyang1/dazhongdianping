# 分类、城市、商圈基础数据治理设计

> **状态：** 用户已确认建设分类、城市、商圈完整治理闭环；本文锁定实现边界，供后续计划和代码验收使用。

## 目标与范围

管理端补齐分类、城市和商圈（`area`）的列表、新增、编辑、排序、启停与受保护删除能力。数据继续按 `CN/EU` 区域隔离，管理员只能操作 `X-Region` 已授权的区域。

本期确认的业务语义：

- 停用项保留在管理端并标明状态；C 端元数据接口和筛选项不再返回停用项。
- 已存在门店不因基础数据停用而自动下线，详情和历史列表仍保留原名称；但新建、编辑、导入和商户门店变更不能选择停用项。
- 被实际业务数据引用的记录不得物理删除，只能停用；未被引用的记录才允许删除。
- 分类保持当前 `parent_id` 树模型，不人为限制层级深度。

本期不实现城市或门店级管理员数据范围、用户治理、运营位维护页面、审计日志查询或历史数据修复任务。当前双 schema 是开发/验收初始化脚本，不在本文虚构生产库滚动迁移能力。

## 现状依据

- `category`、`city`、`area` 当前只有 ID、区域、名称、排序和层级/归属字段，没有状态字段，也没有可用于新增的自增 ID。
- C 端已通过 `/api/c/v1/categories`、`/cities`、`/cities/{cityId}/areas` 读取这些数据；分类树、搜索联想和热门词也直接读取 `category`。
- `shop`、`merchant_shop_change`、`rank_config`、`rank`、`home_banner` 与 `home_feed` 已保存相应 ID，不能把基础数据当成一张随手可清空的字典表。
- 管理端已有数据库 RBAC、`@AdminPermission`、菜单服务、路由守卫和 `data:shop:*` 权限。基础数据治理必须使用新的对象权限，不借用门店权限扩大授权范围。

## 方案选择

采用“在原基础表直接增加状态和约束 + 独立管理模块”的方案。

未采用独立状态覆盖表：它会让公共查询、管理查询和写入校验都变成双表拼装，数据语义反而更散。也未采用只做软删除的方案：软删除无法区分暂时下线与已移除，无法满足恢复和历史可见的治理需要。

## 数据模型与一致性

### 表变更

H2 `backend/src/main/resources/schema.sql` 与 MySQL `sql/mysql/01_schema.sql` 同步调整：

| 表 | 新增/调整 | 约束 |
|---|---|---|
| `category` | `id` 改为自增；新增 `status TINYINT NOT NULL DEFAULT 1` | 同一区域、同父分类下名称唯一；查询索引覆盖 `region/status/parent_id/sort_no/id` |
| `city` | `id` 改为自增；新增 `status TINYINT NOT NULL DEFAULT 1` | `region + code` 唯一，`region + name` 唯一；查询索引覆盖 `region/status/sort_no/id` |
| `area` | `id` 改为自增；新增 `status TINYINT NOT NULL DEFAULT 1` | `region + city_id + name` 唯一；查询索引覆盖 `region/city_id/status/sort_no/id` |

既有种子数据显式使用原 ID，默认状态均为启用；自增只服务后续管理端新增，不改变既有引用。`backend/src/main/resources/data.sql` 与 `sql/mysql/02_seed_data.sql` 同步维护新增权限和角色授权，避免 H2 与 MySQL 的 RBAC 种子漂移。

### 层级和状态规则

- 所有名称和城市编码先 trim；城市编码统一转为大写后再参与唯一性校验。`category.parent_id = 0` 表示根分类。新增或换父分类时，父分类必须存在、属于当前区域且启用；禁止指向自身或任一后代，防止环。
- 分类若有启用的直接子分类，不得停用；分类若有任意子分类，不得删除。子分类只有父分类启用时才可启用。
- 商圈新增、编辑或启用时，所属城市必须存在、属于当前区域且启用。
- 城市停用不级联修改商圈；C 端按城市启用状态过滤商圈，因此恢复城市后原有启用商圈可恢复可见。
- 所有状态变更和删除只查询当前区域中的记录，不能通过传 ID 跨区碰运气。

不在本期新增跨表外键。现有 schema 本就以应用层区域校验和引用保护为主，直接补一组不一致的物理外键会使 H2/MySQL 初始化和历史种子行为分叉；本期用事务内校验保证治理行为一致。

## 后端架构

新增 `module/admin/geodata`，与庞大的门店管理服务分开：

- `AdminGeoDataController`：管理端 HTTP 契约和 `@AdminPermission`。
- `AdminGeoDataService`：区域、层级、状态、重复性和引用保护规则。
- `AdminGeoDataMapper` 及 XML：元数据 CRUD、活跃性校验、引用计数和排序读取。
- request/response/model：分别表达分类、城市、商圈，避免把三种不同约束塞进一个万能 DTO。

写操作使用事务。Bean Validation 和字段格式错误返回 `400`；不存在或不在当前区域的记录返回 `404`；重复名称/编码、树结构、状态和引用冲突使用现有 `ConflictException` 返回 `409`；权限或区域范围不足仍由既有拦截器返回 `403`。前端因此不必从模糊 `500` 猜原因。

### 管理端接口

接口统一位于 `/api/admin/v1`，区域从已授权的 `X-Region` 获取，payload 不接受可伪造的 `region` 字段。

| 资源 | 接口 | 权限 | 说明 |
|---|---|---|---|
| 分类 | `GET/POST /categories`、`PUT/DELETE /categories/{id}`、`PUT /categories/{id}/status` | `data:geo:read/write` | 管理列表返回启用与停用项及树关系 |
| 城市 | `GET/POST /cities`、`PUT/DELETE /cities/{id}`、`PUT /cities/{id}/status` | `data:geo:read/write` | 管理列表返回启用与停用项 |
| 商圈 | `GET/POST /areas`、`PUT/DELETE /areas/{id}`、`PUT /areas/{id}/status` | `data:geo:read/write` | `GET` 使用 `cityId` 过滤，即使该城市已停用也允许管理员查看其商圈；写操作校验城市归属 |

读取方法要求 `data:geo:read`，其余方法要求 `data:geo:write`。所有基础数据接口保持 `regionScoped=true`，由现有管理员区域范围继续保护。

### 公开端与现有写入链路

公共 `BrowseQueryMapper` 和相关服务统一只读取启用项：

- 分类树、分类搜索联想和分类热门词排除停用分类。
- 城市列表排除停用城市；商圈查询同时要求自身与所属城市启用。
- 使用停用分类、城市或商圈 ID 的 C 端门店筛选返回空分页，避免绕过筛选 UI 继续暴露已下线元数据。
- 门店详情与没有元数据筛选条件的既有门店列表继续保留历史分类、城市、商圈名称；门店自身是否公开仍只由现有门店状态控制。

管理端门店新增、编辑、导入以及商户的门店变更申请，均将现有“存在且区域匹配”校验改为“启用且区域/归属匹配”。商户变更审核落库前必须重新校验，避免草稿保存后基础数据被停用仍写入门店。榜单草稿创建、编辑与发布前也重新校验城市和分类启用状态，避免草稿在发布窗口继续生成新引用。这样停用是可恢复的治理动作，却不会再产生新的脏引用。

## 引用保护

删除在同一事务内统计当前区域的所有已落库引用，任一计数大于零即拒绝。检查范围：

| 资源 | 结构/业务引用 |
|---|---|
| 分类 | 子分类、`shop`、`merchant_shop_change`、`rank_config`、`rank` |
| 城市 | `area`、`shop`、`merchant_shop_change`、`home_banner`、`home_feed`、`rank_config`、`rank` |
| 商圈 | `shop`、`merchant_shop_change` |

引用计数不因门店软删、变更单状态或运营位启用状态而忽略，历史记录仍应保留可解释的基础数据 ID。删除成功后不复用 ID。

## 权限、菜单与管理端页面

新增两个种子权限：

- `data:geo:read`：查看基础数据。
- `data:geo:write`：维护基础数据。

`super_admin` 获得全部当前权限；`data_operator` 获得这两个基础数据权限。菜单服务在“数据管理”组新增“基础数据”叶子，路由为 `/data/meta`，并使用 `data:geo:read` 作为前端守卫条件。后端权限注解仍是唯一安全边界。

新增 `BasicDataManagementView`：

- 顶部显示当前区域上下文，区域切换继续走已有全局机制，不在表单内伪造区域选择器。
- 以分类、城市、商圈三个页签切换工作区。分类使用紧凑树表；城市与商圈使用可筛选表格，商圈先选择城市。
- 新增/编辑使用同一表单状态；排序为数字输入；启停为明确的二次确认动作；删除按钮只在拥有写权限时显示。
- 管理列表保留停用行并显示状态。删除受阻时展示后端返回的引用原因，不自行猜测引用关系。
- 路由守卫和按钮可见性都从实时 `auth/me` 权限集刷新，角色停用后的既有 token 按现有 RBAC 语义实时失去可操作能力。

## 非目标

- 不提供管理员按城市/门店的细粒度授权。
- 不新增运营位、榜单或审计日志的管理页面；它们仅作为删除保护的已有引用源。
- 不批量迁移、合并或重分配已有基础数据。
- 不改变 C 端门店自身的上下架规则，也不因元数据停用自动下线既有门店。

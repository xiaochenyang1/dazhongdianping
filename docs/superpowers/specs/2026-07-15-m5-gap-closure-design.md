# M5 仓库内缺口收口设计

> 日期：2026-07-15

## 目标

在不伪造外部云凭证、真实支付资质或 Flutter 交付物的前提下，收口当前仓库明确标记为未完成的 M5 工程项：独立商户工作台、管理端商户点评申诉专页和 PC Web 实时通知。完成后用自动化测试与构建结果证明，而不是只更新文档口径。

## 范围边界

- 新建独立 `merchant-web`，复用现有 `/api/b/v1` 登录、账号、门店、预订、团购、订单、草稿、点评与申诉接口。
- 在 `admin-web` 增加商户点评申诉专页，仅展示并处理 `bizType=6` 审核任务。
- 后端新增持久化通知、未读计数、已读确认、WebSocket ticket 和实时推送；`web` 接入通知中心与断线重连。
- M6 Flutter、真实 Stripe/支付宝/短信邮件、Google Maps 和 M7 社区社交属于独立阶段；本轮只校验文档没有把它们误报成已完成，不用空壳冒充交付。

## 架构

### 商户工作台

采用 Vue 3 + TypeScript + Vite，目录结构与 `admin-web` 保持一致。会话存储、区域头和错误处理集中在 composable/http 层；页面只负责查询条件、表单状态和业务动作。首个可交付版本覆盖登录、概览、门店、预订、团购、订单、点评/回复/申诉，所有页面访问都经过商户登录守卫。

### 管理端申诉页

复用 `AdminAuditTask` 和审核通过/驳回 API，通过固定查询参数 `bizType=6` 隔离商户点评申诉。页面提供状态筛选、分页、理由展示、通过备注和驳回原因，处理后刷新当前页。

### 通知

通知先写数据库，再尝试向当前在线用户推送。REST 接口负责列表、未读数和已读确认，WebSocket 只承担低延迟分发；前端断线或未在线时，重新进入页面仍可从数据库补齐。ticket 为短时一次性凭证，避免把 access token 暴露到 WebSocket URL。

## 错误处理与安全

- 所有 B 端操作继续由后端 RBAC 与门店范围兜底，前端隐藏按钮不能当权限控制。
- WebSocket ticket 绑定用户、区域和过期时间，消费后失效。
- 通知确认使用所属用户条件更新，禁止跨用户 ACK。
- 页面统一显示 API 错误；业务动作成功后刷新数据，避免前端自行猜测服务端终态。

# M5b3 商户门店资料、相册与菜单管理设计

## 1. 目标与范围

本阶段完成商户端门店基础资料、相册和菜单/推荐菜管理，并遵守“草稿审核通过后整体替换线上数据”的生效策略。商户编辑过程不得直接修改正在营业的 `shop`、`shop_photo` 和 `dish` 数据，审核驳回或审核期间的半成品也不得影响 C 端浏览、搜索、交易、预订和经营看板。

本阶段包含：

- 商户创建新门店变更草稿。
- 商户从已有门店复制完整快照，编辑基础资料、相册和菜单。
- 草稿保存、详情查询、列表查询和提交审核。
- 同一草稿内分别维护门店基础信息、照片顺序和菜品顺序。
- 管理端在统一审核中心通过或驳回门店变更。
- 审核通过后以单事务创建新门店或整体替换已有门店资料。
- 审核通过后增量同步 Elasticsearch；失败时保留管理端全量重建入口作为补偿。
- 商户与管理端关键动作写入现有操作日志和审计日志。

本阶段不包含独立 `merchant-web` 页面、门店认领纠纷、连锁品牌模型、营业时间结构化排班、批量菜单导入和图片内容机审。这些能力分别留给 M5c、后续门店治理或内容审核阶段。

## 2. 方案选择

采用独立变更申请快照，不直接给线上 `shop` 增加“半完成编辑态”。线上表继续只表达当前已审核、可被业务读取的门店数据；草稿表表达尚未生效的完整候选版本。

不采用“修改即覆盖线上表后下架”的方案，因为它会让审核期间的门店从公开浏览、交易和预订链路消失。不采用“基础信息、相册、菜单分别审核”的方案，因为三个审核任务可能只通过一部分，形成封面、菜品和门店描述相互不匹配的拼装状态。

## 3. 数据模型

### 3.1 `merchant_shop_change`

保存一份可独立审核的门店候选版本：

- `id`：变更申请 ID。
- `merchant_id`、`operator_id`、`region`：提交主体、实际操作者和区域。
- `change_type`：`1新建门店 2修改门店`。
- `target_shop_id`：修改门店时指向线上门店；新建草稿审核通过前为 `0`。
- `base_updated_at`：创建修改草稿时记录线上 `shop.updated_at`，用于审核通过前的乐观并发校验。
- 门店候选字段：`category_id`、`city_id`、`area_id`、`name`、`cover_url`、`phone`、`price_per_capita`、`currency`、`address`、`latitude`、`longitude`、`business_hours`、`summary`、`open_now`、`tags`。
- `status`：`0草稿 1待审核 2通过 3驳回 4已失效`。
- `reject_reason`、`audit_by`、`submitted_at`、`audited_at`、创建和更新时间。

评分、点评数和 `has_deal` 不允许由商户填写。新门店审核通过时评分初始化为 `0`，点评数为 `0`，`has_deal=false`；修改门店时继续保留线上聚合值。

### 3.2 `merchant_shop_change_photo`

保存候选相册快照：`change_id`、`image_url`、`photo_type`、`sort_no`。一期 `photo_type` 支持 `1门店图 2环境图 3菜单图`。每个草稿最多 20 张图片，排序号在草稿内稳定。

### 3.3 `merchant_shop_change_dish`

保存候选菜单/推荐菜快照：`change_id`、`name`、`price`、`recommend_reason`、`sort_no`。每个草稿最多 100 个菜品，价格不能为负数。

### 3.4 线上照片与菜品主键

现有 H2/MySQL 的 `shop_photo.id` 和 `dish.id` 改为自增主键。种子脚本仍可显式写入固定 ID；审核通过时由数据库为新快照生成 ID，避免服务层使用 `MAX(id)+1` 产生并发冲突。

## 4. 商户端接口

所有接口使用 `/api/b/v1` 前缀，要求商户登录且资质审核通过。

### 4.1 草稿列表与详情

- `GET /shop-changes`
  - 参数：`shopId`、`status`、`changeType`、`page`、`pageSize`。
  - 权限：`shop:edit`。
  - 只返回当前商户、当前区域和员工门店范围内的申请。
- `GET /shop-changes/{changeId}`
  - 返回完整基础资料、照片、菜品、审核状态和驳回原因。
  - 修改草稿必须继续校验目标门店范围。

### 4.2 创建草稿

- `POST /shops/change-drafts`
  - 创建新门店草稿，`change_type=1`、`target_shop_id=0`、`status=0`。
- `POST /shops/{shopId}/change-drafts`
  - 权限：`shop:edit`，并校验目标门店归属与员工门店范围。
  - 复制线上基础资料、照片和菜单，记录 `base_updated_at`，生成 `change_type=2` 草稿。

同一商户可同时维护多个新门店草稿；同一线上门店只允许一个 `status IN (0,1)` 的有效草稿。创建新草稿时若已有有效草稿，返回现有草稿摘要而不是静默再造一份。

### 4.3 编辑草稿

- `PUT /shop-changes/{changeId}`：保存基础资料。
- `PUT /shop-changes/{changeId}/photos`：整体替换候选相册。
- `PUT /shop-changes/{changeId}/dishes`：整体替换候选菜单。

只有 `status=0` 或 `status=3` 的申请可编辑。编辑被驳回申请时，服务先将其重置为草稿并清空旧审核信息；已通过、待审核或已失效申请不可编辑。

### 4.4 提交审核

- `POST /shop-changes/{changeId}/submit`
  - 校验基础资料、分类/城市/商圈、区域币种、相册、菜单和门店范围。
  - 将申请原子更新为 `status=1`，创建 `audit_task.biz_type=5` 的待审核任务。
  - 同一申请重复提交返回状态冲突，不重复创建审核任务。

提交时至少需要一张门店图；`cover_url` 必须等于候选相册中某一张图片，防止封面和相册脱节。

## 5. 管理端审核

统一复用：

- `GET /api/admin/v1/audit/tasks?bizType=5`
- `POST /api/admin/v1/audit/tasks/{taskId}/pass`
- `POST /api/admin/v1/audit/tasks/{taskId}/reject`

审核列表对 `biz_type=5` 返回变更类型、候选门店名、商户名称、目标门店和摘要。

### 5.1 审核通过新建门店

在同一事务内：

1. 原子更新审核任务 `0 -> 1`。
2. 原子更新变更申请 `1 -> 2` 并写审核人/时间。
3. 新建 `shop`，强制使用申请的商户和区域，评分/点评/团购聚合值使用安全初始值。
4. 批量写入 `shop_photo` 与 `dish`。
5. 回填 `target_shop_id`。
6. 为商户全门店范围主账号自动可见；指定门店员工不会自动获得新门店权限，由主账号后续授权。
7. 写管理端审计日志。

### 5.2 审核通过修改门店

在同一事务内：

1. 锁定目标门店，校验商户、区域和未删除状态。
2. 比较当前 `shop.updated_at` 与申请的 `base_updated_at`；不一致时拒绝应用并提示线上资料已变化。
3. 原子更新审核任务和变更申请状态。
4. 只替换商户可编辑字段，保留评分、点评数、团购状态等平台聚合字段。
5. 删除并重建该门店的照片和菜品快照。
6. 写管理端审计日志。

事务提交后发布 `ShopSearchIndexChangedEvent`。ES 同步异常不回滚数据库事务，记录错误并由现有管理端重建索引接口补偿。

### 5.3 审核驳回

审核任务与申请分别原子更新为驳回状态，保存原因和审核人。线上门店、照片、菜单和搜索索引完全不变。商户可编辑该驳回申请并重新提交，重新提交时生成新的待审核任务，旧任务保留历史状态。

## 6. 校验、权限与错误处理

- 商户接口缺少权限返回 `401`。
- 非当前商户、错区域或超出员工门店范围统一返回 `404`，避免泄露门店和草稿存在性。
- 非法分类、城市、商圈、币种、经纬度、金额、图片数量、菜品数量和状态迁移返回 `400`。
- 同一门店存在有效草稿时不创建重复草稿。
- 线上版本已变化、重复审核、重复提交或旧审核任务返回 `400` 状态冲突。
- 标题、地址、电话、营业时间和摘要长度与当前 `shop` 字段容量一致；所有文本入库前去除首尾空白。
- `CN` 草稿币种固定为 `CNY`；`EU` 当前固定为 `EUR`，后续多币种在 M6 扩展。
- 创建和应用草稿的 SQL 必须同时限定 `merchant_id`、`region`、状态和未删除标记。

## 7. 数据流

修改门店的正常链路：

1. 商户从线上门店创建草稿快照。
2. 商户多次保存基础资料、照片和菜品，C 端始终读取线上版本。
3. 商户提交后草稿冻结，审核中心出现单个门店变更任务。
4. 管理员通过后事务整体替换线上数据。
5. 提交事务后同步 ES，C 端后续请求读取新版本。

任何失败都不得形成“基础资料已更新但照片/菜单没更新”的部分状态。

## 8. 测试与验收

严格执行 RED → GREEN：

- 新建门店草稿未通过前，公开门店列表和详情不可见。
- 修改草稿保存和待审核期间，公开门店仍返回旧资料、旧相册和旧菜单。
- 审核通过新建门店后，基础资料、照片、菜单完整可见且属于正确商户/区域。
- 审核通过修改门店后，三个线上表一次性替换；评分、点评数和团购聚合字段不被商户快照覆盖。
- 审核驳回不改变任何线上数据，驳回原因对商户可见，修改后可重新提交。
- 审核期间线上门店被管理员修改时，旧草稿因版本冲突不能覆盖新数据。
- 店长只能处理授权门店；跨商户、错区和无权限访问按约定拦截。
- 分类/城市/商圈、币种、图片数量、菜品数量和封面一致性校验完整。
- 审核任务重复处理、草稿重复提交和同门店重复有效草稿被拦截。
- H2 `schema.sql`、MySQL `01_schema.sql` 和种子数据保持一致。
- 管理端审核列表兼容点评、团购和门店变更三类任务。
- 审核通过后发布搜索索引同步事件。

阶段完成后依次执行定向测试、`backend\\mvnw.cmd test` 和根目录 `./scripts/ci/verify-all.ps1`。

## 9. 自审结论

- 没有 `TBD`、`TODO` 或未定义状态。
- 草稿、提交、审核、应用和重新提交的状态迁移完整。
- 新建与修改共用同一套变更申请模型，但应用路径明确分开。
- 线上聚合字段与商户可编辑字段边界明确。
- 基础资料、相册和菜单作为一个快照审核，不存在部分生效。
- 权限、区域、门店范围、版本冲突、日志和 ES 补偿均有验收项。

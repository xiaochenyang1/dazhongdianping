# M3b 榜单设计

## 目标

交付按区域、城市、分类隔离的必吃榜、好评榜、热门榜；同时提供运营规则的草稿、版本发布、重算快照和回滚能力。公开接口只读取已发布快照，重算失败时继续保留旧快照。

## 数据模型

- `rank_config`：版本化规则。`status=0` 为草稿，`status=1` 为已发布，`status=2` 为已归档；同一作用域只能有一个已发布版本。
- `rank`：一次成功发布生成的不可变榜单快照，记录命中的 `config_id` 和周期。
- `rank_item`：快照内的门店排名、计算分和入榜理由。
- 发布事务先完成规则校验和榜单计算，再写入新快照并切换已发布版本；任何一步失败都回滚，旧快照不受影响。
- 回滚不修改历史快照，而是将目标历史规则复制为一个新版本并重新发布，确保版本时间线只向前走。

## 计算规则

- 必吃榜：综合评分、点评量、优惠状态和人工加权。
- 好评榜：综合评分、点评量与人工加权。
- 热门榜：点评量、优惠状态、营业状态与人工加权。
- `weight_json` 保存各指标权重；缺少订单量、收藏量等尚未落地的数据源时拒绝配置非零权重，不能拿常量伪造。
- 先过滤区域、城市、分类、营业状态、最低点评量和最低评分，再计算并取前 50 名。

## 接口

- 公开：`GET /api/c/v1/ranks`、`GET /api/c/v1/ranks/{id}`。
- 管理：`GET/POST/PUT /api/admin/v1/ranks/config`、`POST /api/admin/v1/ranks/config/{id}/publish`、`POST /api/admin/v1/ranks/config/{id}/rollback`。
- 管理写接口继续使用管理员鉴权和 `X-Region` 隔离。

## 缓存与一致性

- 第一版以 MySQL 快照为权威源，接口结构预留缓存层；发布成功后发出榜单发布事件，后续 Redis 缓存监听该事件失效。
- 不在请求时现算榜单，避免公开接口结果抖动和数据库被拖垮。

## 验收

- 三类榜单均能按城市与分类查询，详情排名稳定。
- CN/EU 不串区。
- 草稿不影响线上；发布生成新快照；非法规则发布失败且旧榜单仍可读；回滚生成新版本。
- H2、MySQL schema 与 seed 同步，后端测试和全量 CI 通过。

## 1. 文档目标

统一名词、命名和枚举。文档最怕啥? 不是少写两页,是同一个东西今天叫 `shop`,明天叫 `merchant`,后天又冒出个 `poi` 装新鲜。

## 2. 核心术语

| 术语 | 英文标识 | 含义 | 备注 |
|---|---|---|---|
| 用户 | `user` | C 端普通账号 | 可跨区域有行为数据 |
| 商户主体 | `merchant` | B 端入驻主体 | 可管理多个门店 |
| 门店/商户展示实体 | `shop` | C 端展示与交易主体 | POI 级对象 |
| 分类 | `category` | 门店所属类目 | 可多级 |
| 城市 | `city` | 区域下城市 | 受 `region` 约束 |
| 商圈/行政区 | `area` | 城市下地理分组 | 文本筛选主体 |
| 团购/代金券产品 | `deal` | 可售卖的优惠产品 | 不等于券码 |
| 券码 | `coupon` | 支付成功后生成的可核销实例 | 一个订单可多张 |
| 订单 | `order` | 下单支付主体 | 与支付、退款关联 |
| 预订 | `reservation` | 到店前预约记录 | 与团购订单并列 |
| 点评 | `review` | 用户对门店的评价 | 需审核 |
| 帖子 | `post` | 社区内容 | APP 为主 |
| 话题 | `topic` | 帖子聚合标签 | 热榜可配置 |
| 圈子 | `group` | 社区内用户分组 | 对外接口叫 `group`,数据库内部叫 `circle` |
| 通知 | `notification` | 站内消息持久化记录 | 可推送可补拉 |
| 设备 | `user_device` | 用户终端与 push token 记录 | APP 推送依赖 |
| 商户员工 | `merchant_staff` | 商户主账号创建的员工登录主体 | 按角色与门店范围授权 |
| 运营活动 | `operation_activity` | 后台配置的专题/活动主体 | 挂资源项对外展示 |
| 隐私导出任务 | `privacy_export_task` | 用户发起的数据导出申请 | 异步生成下载包 |
| 隐私删除任务 | `privacy_delete_task` | 用户发起的删除/注销申请 | 含冷静期与匿名化 |

## 3. 命名规范

### 3.1 API 命名

- 路径统一用英文复数名词,例如 `/shops`、`/orders`。
- 动作型接口尽量挂在资源下,例如 `/orders/:id/pay`。
- 管理端和商户端通过前缀区分,不要再额外发明奇怪命名。

### 3.2 数据表命名

- 表名统一单数,例如 `shop`、`review`、`reservation`。
- 关联表用清晰组合名,例如 `post_topic`、`admin_role`。
- 状态字段统一 `status` 或业务前缀状态,别同一类对象整出三四个随意命名。

## 4. 区域语义

当前文档里 `region` 被拿来表达好几层意思,现在统一按下面理解:

| 概念 | 含义 | 当前落点 |
|---|---|---|
| `request_region` | 当前请求上下文区域 | Header / Query |
| `data_region` | 业务数据所属区域 | 商户、团购、点评、帖子等表字段 |
| `preferred_region` | 用户默认浏览偏好 | `user.preferred_region` |
| `management_scope` | 管理员/审核员可操作的区域范围 | 权限模型/数据权限 |

补充约束:

- `merchant.region` = 商户主体经营区域,一期单主体单区域。
- `preferred_region` 不参与访问控制,只做偏好与后续运营策略。

## 6. 已拍板口径

### 6.1 收藏目标类型

一期只保留 `1店铺 2帖子`。若后续要支持团购/点评收藏,必须新增枚举值,不能复用现有值硬拧。

### 6.2 预订状态

已区分“用户取消”和“商户拒绝”,并采用 `0-5` 六态口径。

### 6.3 圈子命名

- 对外接口统一叫 `group`。
- 数据库内部继续使用 `circle`,避免与 SQL 保留字冲突。

### 6.4 私信举报对象

`report.target_type` 已扩展覆盖私信消息和私信会话。

### 6.5 删除/注销二次校验

账号删除/注销统一使用 `verification_code.scene=delete` 做验证码二次确认,别复用 `reset` 这种半吊子语义。

| `0` | 成功 |

| `3000-3999` | 点评与 UGC |

| `5000-5999` | 社区与社交 |

## 8. 使用规则

- 新增实体前先补术语表。
- 新增枚举值前先补本文件,再改接口和库表。
- 同一字段在不同文档里含义不一致时,先停下来改文档,别拿代码当裁判。

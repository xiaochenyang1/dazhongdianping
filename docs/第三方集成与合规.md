# 大众点评(仿)项目 · 第三方集成与合规

> 版本:v0.3(补齐支付回调验签、渠道幂等与隐私权利收口)
> 最后更新:2026-07-09
> 配套文档:`需求文档.md`、`系统架构设计.md`、`接口设计.md`、`业务流程与状态机.md`、`第三方供应商选型表.md`

## 1. 文档目标

这份文档管两类事:

- 对外接了哪些第三方,接在哪,有什么安全边界。
- 国内区和欧洲区在支付、隐私、税务、推送这些地方分别要遵守什么规则。

## 2. 接入原则

- 所有第三方通过适配层封装,业务层不直接写死供应商 SDK。
- 每个回调都必须验签、幂等、可重放补偿。
- 所有密钥按环境隔离:本地、测试、预发、生产。
- 涉及用户隐私或交易的回调都必须进审计日志。

## 4. 关键接入点

### 4.1 账号验证

- 邮箱验证码
- 短信验证码
- 找回密码/换绑验证

当前采用的设计口径:

- 验证码按 `scene + target + deviceId + IP` 做限流。
- 验证码记录落 `verification_code` 表,只存哈希,不存明文。
- 登录、注册、换绑、找回密码统一走同一套验证码能力,通过 `scene` 区分。
- 单设备高频触发时直接拒绝发送,避免被人拿短信/邮件服务当提款机。

### 4.2 支付与退款

- 下单时根据 `region` 和币种选择支付渠道。
- 支付回调必须支持重复通知。
- 退款必须保存原始请求和结果,便于对账。

固定规则:

- 幂等键优先使用 `order_no + channel + channel_txn`。
- 支付成功以服务端异步回调为准,前端同步结果只做展示。
- 回调验签失败、金额不匹配、重复入账都必须打审计日志并拒绝状态变更。
- 退款结果同样要保留原始报文,用于人工对账和补偿。

分渠道接入要点:

- **Stripe**:
  - 回调头:`Stripe-Signature`
  - 必须用**原始 body** + endpoint secret 验签。
  - 订单号建议放 `PaymentIntent.metadata.orderNo`,别指望前端回跳兜底。
- **PayPal**:
  - 回调头至少包含 `PAYPAL-TRANSMISSION-ID / TIME / SIG / AUTH-ALGO / CERT-URL`。
  - 必须调 PayPal 验签接口做二次校验,同时本地校验 `resource.custom_id`、金额、币种。
- **支付宝**:
  - 异步通知一期按表单参数处理,核心字段是 `out_trade_no / trade_no / trade_status / total_amount`。
  - 只认 `TRADE_SUCCESS / TRADE_FINISHED`,处理成功返回纯文本 `success`。
- **微信支付 v3**:
  - 回调头使用 `Wechatpay-*`,先验签再解密 `resource.ciphertext`。
  - 解密后核心字段是 `out_trade_no / transaction_id / trade_state / amount.total`。

发起支付 / 退款时的渠道幂等建议:

- Stripe 请求头带 `Idempotency-Key`。
- PayPal 请求头带 `PayPal-Request-Id`。
- 支付宝、微信支付用商户侧 `out_trade_no` / 退款单号做唯一约束。

### 4.3 地图与定位

- PC 端当前不做地图 UI。
- APP 欧洲区使用 Google Maps。
- 地理坐标仍作为商户数据的基础字段保存,用于距离排序和导航。

### 4.4 推送与消息

- PC 实时消息走 WebSocket。
- APP 离线消息走 FCM / APNs。
- `user_device` 记录设备、平台、push channel、push token、状态、最后活跃时间。
- 登录后注册设备,token 刷新时更新,登出时停用 token 但保留设备记录。
- 连续推送失败的 token 自动停用,避免死循环重试。

### 4.5 文件与媒体

- 图片上传建议走服务端签名 + 客户端直传。
- OSS/S3 按区域分桶。
- URL 出口统一由文件服务生成,别让业务代码到处拼 CDN 域名。

## 5. 合规基线

### 5.1 EU 区

- GDPR:数据最小化、用途告知、删除/导出能力、隐私政策留痕。
- SCA / 3D Secure:支付认证流程必须跟上。
- VAT:价格展示、税费说明、账单/发票策略要清楚。
- 数据导出任务建议默认 **7 天下载有效期**,下载文件加签名 URL。
- 数据删除默认进入 **7 天冷静期**,期间允许用户撤销。

### 5.2 CN 区

- 隐私政策、用户协议、Cookie/日志告知要到位。
- 图片、评论、社区内容需要符合内容审核要求。
- 支付、退款、发票等交易留痕不能少。

### 5.3 跨区共性

- 用户注销后,需要明确哪些数据删除、哪些做匿名化保留。
- 用户数据导出要能定位到订单、点评、帖子、私信等核心对象。
- 审计日志与交易流水保留期限要单独约定。
- 协议 / 隐私政策确认必须留痕到版本级,别只在页面上弹一下就装合规了。

## 6. 安全要求

- Webhook 验签。
- 幂等键或回调去重。
- IP 白名单或来源校验。
- 敏感配置只存环境变量或密钥管理服务。
- 第三方失败时有降级与重试策略。
- 回调原文必须保留到审计日志或对象存储,便于复盘签名争议。
- 退款、删除任务、数据导出都要有二次确认和操作留痕。

## 7. 建议补齐的配置项

- `STRIPE_*` Webhook secret、API key、价格/货币开关。
- `PAYPAL_*` Client ID、Secret、Webhook ID、环境切换。
- `ALIPAY_*` appId、商户私钥、支付宝公钥、sellerId。
- `WECHATPAY_*` mchid、商户证书序列号、APIv3 key、平台证书。
- `MAIL_*` 邮件服务配置。
- `SMS_*` 短信服务配置。
- `MAP_*` 地图密钥配置。
- `PUSH_*` 推送服务配置。
- `STORAGE_*` 存储桶与 CDN 配置。
- `CONTENT_AUDIT_*` 审核服务配置。
- `PRIVACY_*` 冷静期天数、导出文件有效期、删除保留策略。

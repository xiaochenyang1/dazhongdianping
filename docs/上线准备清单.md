## 1. 文档目标

这份文档就是上线前的总闸门。它不负责解释业务理想,只负责防止上线当天出现这些经典事故:

- 环境变量少配一条,启动就炸。
- 回调地址填错一个字母,支付全丢。
- 证书过期没人看,CDN 直接红锁。
- 监控没配,线上已经着火了,群里还在问“谁有空看一下”。

## 2. 使用方式

上线前按这个顺序走:

1. 先核对环境与依赖。
2. 再核对第三方和回调。
3. 再做预发回归和上线演练。
4. 最后执行生产上线与冒烟。

每一项都建议有负责人和完成时间,别全写成“待确认”这种废话。

## 11. 灰度策略

一期建议别玩太花,按最朴素的来:

- 先 `pre` 全量回归
- 再生产小流量灰度
- 再逐步放量

建议灰度维度:

- 按内部账号白名单
- 按城市或区域
- 按接口开关

## 12. 回滚策略

### 12.1 触发条件

- P0 链路异常
- 支付回调异常率明显升高
- 登录、验证码、审核、导出任务大面积失败
- 数据串区或严重数据错乱

### 12.2 回滚动作

1. 立即停止继续放量。
2. 切回上一个稳定版本。
3. 回退配置变更。
4. 必要时回退数据库脚本或执行补偿脚本。
5. 公告相关负责人和业务方。

### 12.3 回滚后检查

- [ ] 首页/列表/详情恢复
- [ ] 登录恢复
- [ ] 回调恢复
- [ ] 监控恢复稳定
- [ ] 缺陷单和复盘记录创建

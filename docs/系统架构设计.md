# 大众点评(仿)项目 · 系统架构设计

> 版本:v0.3(补齐运营配置、幂等限流与隐私中心架构口径)
> 最后更新:2026-07-09
> 配套文档:`需求文档.md`、`接口设计.md`、`数据库设计.md`、`业务流程与状态机.md`、`第三方集成与合规.md`

## 1. 文档目标

本文只回答系统级问题:

- 系统由哪些端和哪些核心模块组成。
- 关键数据流怎么走,哪里同步、哪里异步。
- `CN` / `EU` 区域如何隔离,哪些能力跨区域统一。
- 哪些基础设施是刚需,哪些是一期可后补。

不负责字段级接口和 DDL 细节,那些去看配套文档。

## 2. 架构原则

- 一套后端服务体系承接 C 端、B 端、管理端,通过前缀和角色隔离。
- 业务数据按 `region` 隔离,账号体系、鉴权、管理能力跨区域统一。
- MySQL 是权威数据源,Elasticsearch 是搜索与排序引擎,Redis 是缓存与会话层。
- 先支持单体式模块化架构,后续数据量上来再做服务拆分或热点拆库。
- 所有对外集成必须可替换,别把支付、推送、短信直接写死在业务层里。

## 3. 系统上下文

### 3.1 四类前端

- C 端 PC Web:国内浏览、搜索、点评、榜单、交易。
- C 端 APP:欧洲华人本地生活 + 社区社交 + 地图 + 海外支付。
- B 端商户工作台:商户主体、门店、团购、核销、经营数据。
- 管理端:审核、运营配置、基础数据、RBAC、审计。

### 3.2 核心后端模块

| 模块 | 主要职责 | 直接依赖 |
|---|---|---|
| 认证与账号 | 注册登录、JWT、绑定、账号安全、验证码 | MySQL、Redis、邮件/短信 |
| 用户中心 | 个人资料、成长值、等级、收藏 | MySQL、Redis |
| 商户与地理 | 门店、分类、城市、商圈、相册、菜单 | MySQL、对象存储 |
| 搜索服务 | 联想、搜索、筛选、排序、热词 | ES、Redis、MySQL |
| 点评与 UGC | 点评、评论、点赞、举报、审核流 | MySQL、对象存储、审核服务 |
| 交易服务 | 团购、订单、支付、券码、退款 | MySQL、支付网关、消息队列 |
| 预订服务 | 时段、预订、确认、改期、爽约、提醒 | MySQL、消息队列、通知服务 |
| 社区社交 | 帖子、关注、私信、圈子、消息 | MySQL、Redis、WebSocket、推送 |
| 内容运营 | Banner、榜单、推荐位、热词、活动、规则发布 | MySQL、Redis |
| 隐私中心 | 数据导出、删除、匿名化、协议留痕 | MySQL、对象存储、消息队列 |
| B 端商户服务 | 入驻、员工角色、门店管理、核销、经营看板 | MySQL、对象存储 |
| 管理后台 | 审核、RBAC、审计、运营配置 | MySQL、Redis |
| 文件服务 | 上传签名、对象存储、CDN URL 管理 | OSS/S3、CDN |

## 4. 区域与边界

### 4.1 区域对象

- `request_region`: 当前请求的浏览/操作区域。
- `data_region`: 商户、点评、团购、榜单等业务数据所属区域。
- `account_scope`: 账号可访问的区域范围。
- `management_scope`: 运营/审核员可管理的区域范围。

### 4.2 当前建议

- C 端当前 `Web` 默认 `CN` 但已支持切换 `CN / EU`,`APP` 规划默认 `EU`,账号本身不应被单一 `region` 限死。
- 商户主体建议先按单区域经营设计;当前 B 端工作台错区请求直接拒绝,后续如要跨区扩成多区域关联表。
- 管理端与审核端必须支持切换区域视角,否则一个后台管两块地就成笑话了。

## 5. 核心数据流

### 5.1 浏览与搜索链路

1. 前端请求公开内容。
2. API 层根据 `request_region`、城市、分类、关键词组装查询。
3. 列表/搜索优先走 ES,详情与交易信息回源 MySQL。
4. 热门榜单、热词、首页运营位优先 Redis 缓存。

### 5.2 点评链路

1. 用户上传图片到对象存储。
2. 前端提交点评正文与图片 URL。
3. 服务层写入 `review` 及媒体表,同步生成审核任务。
4. 审核通过后更新商户聚合评分,并触发搜索索引更新。

### 5.3 团购交易链路

1. 用户创建订单。
2. 服务层锁定库存并生成待支付单。
3. 调第三方支付,等待同步结果和异步回调。
4. 支付成功后生成券码、记录流水、推送通知。
5. 核销、退款、过期通过事件驱动更新订单与券状态。

### 5.4 预订链路

1. 用户先查询门店可订时段,再提交预订。
2. 商户端确认或拒绝,必要时支持改期;自动确认门店可直接跳到已确认。
3. 到店前触发提醒,到店后走统一工作台确认到店。
4. 超时未处理、用户爽约、商户拒绝都要有独立状态和审计痕迹。

### 5.5 社区与通知链路

1. 帖子/评论/私信写 MySQL。
2. 在线用户走 WebSocket,离线用户走推送或下次补拉。
3. 举报与审核任务统一落审核中心,别搞成点评一套、帖子一套、私信再现编一套。

### 5.6 运营配置发布链路

1. 运营后台保存 Banner、榜单规则、活动配置到 MySQL。
2. 草稿发布后生成版本号,同时清理 Redis 热词/榜单/首页缓存。
3. 榜单规则变更触发异步重算任务,生成新的 `rank` / `rank_item` 结果快照。
4. 活动按 `startAt/endAt` 自动切状态,避免运营半夜爬起来手工点上线。

### 5.7 隐私中心链路

1. 用户在设置页发起数据导出或删除申请。
2. 系统做登录态复核、验证码/密码二次确认和频控校验。
3. 导出任务异步聚合订单、点评、帖子、私信等数据后打包到对象存储。
4. 删除任务先进入冷静期,到期后执行 token 失效、设备停用、数据匿名化。
5. 全流程都要落任务状态与审计,这玩意不是弹个确认框就算完事。

## 6. 基础设施与数据角色

| 组件 | 角色 | 一期职责 | 备注 |
|---|---|---|---|
| MySQL | 权威存储 | 业务主数据、交易、审核、权限、配置、隐私任务 | 初期单库 |
| Redis | 高速状态层 | 登录态、验证码、限流、热门缓存、未读数、ws ticket | 需要 TTL 设计 |
| Elasticsearch | 搜索引擎 | 商户/团购/帖子搜索与聚合 | 数据从 MySQL 同步 |
| OSS / S3 | 对象存储 | 图片、导入文件、错误明细文件、隐私导出包 | 按区域分桶 |
| CDN | 加速层 | 图片与静态资源分发 | 国内/海外各自接入 |
| 消息队列 | 异步解耦 | 超时关单、索引同步、审核、通知 | 一期可选,但交易链路最好预留 |
| WebSocket 网关 | 长连接 | PC 实时通知、私信在线消息 | 与登录态打通 |

## 7. 异步任务建议

- 订单超时关闭。
- 支付回调重试与对账补偿。
- 券码生成与过期扫描。
- 预订提醒与超时未确认处理。
- 设备 token 失效清理。
- 榜单重算与缓存刷新。
- 活动定时上线 / 下线。
- 点评/帖子审核。
- 隐私导出包生成与过期清理。
- 冷静期到期后的删除 / 匿名化执行。
- MySQL 到 ES 的索引同步。
- 推送发送与失败重试。

## 8. 非功能要求

### 8.1 一致性

- 交易、库存、券码必须幂等。
- 支付回调必须防重放、防重复入账。
- 验证码、下单、预订、设备注册、隐私任务都要有明确幂等键作用域。
- 审核操作必须可追溯。

### 8.2 安全

- JWT + refresh token。
- 管理端、商户端操作都要审计日志。
- 对外 Webhook 必须验签。
- 上传文件只接受白名单类型与大小限制。
- 数据删除必须二次校验并经过冷静期,不是谁手一抖就把账号扬了。

### 8.3 可观测性

- 所有请求带 `traceId`。
- 关键链路有业务日志和错误告警。
- 支付、审核、导入、推送要有任务维度看板。

## 9. 一期建议部署形态

- 一个后端应用集群,按模块分包,不是上来就把微服务当信仰。
- 一个 MySQL 实例 + 读写备份策略。
- 一个 Redis 集群。
- 一个 ES 集群或托管服务。
- 对象存储与 CDN 按区域分开。
- 后台、商户端、C 端前端各自独立部署。

## 10. 当前未决项

- 搜索同步到底用 canal、应用双写还是定时补偿。
- 商户是否支持跨区域主体经营。
- 私信是走现有通知体系还是独立消息网关。
- 限流规则是否开放后台可配置。
- 运营活动是否要做 AB 实验 / 人群分桶。
- 隐私导出文件是否追加邮件二次确认下载。

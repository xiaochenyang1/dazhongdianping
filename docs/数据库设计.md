## 0. 设计约定(先读这一节)

统一规范,后面所有表都遵守,不再逐张重复说明。

### 0.0 文档定位
- 本文负责表结构、索引、冗余字段、ES 索引概要。
- 字段状态流转、业务动作顺序看 `业务流程与状态机.md`。
- 术语和枚举命名看 `术语与枚举规范.md`。
- 当前尚未拍板但已经暴露冲突的地方,统一看 `文档差异与待确认清单.md`。

**引擎与字符集**
- 引擎:`InnoDB`;字符集:`utf8mb4`;排序规则:`utf8mb4_unicode_ci`(支持 emoji、多语言,配合欧洲区文本)。

**当前代码口径提示**
- 当前仓库可直接导入的 MySQL 初始化脚本见 `../sql/mysql/00_all_in_one.sql`、`../sql/mysql/01_schema.sql` 和 `../sql/mysql/02_seed_data.sql`。
- 这套脚本覆盖当前代码已经实现的 `M1-M4`、隐私中心、`M5a` 商户身份/RBAC、`M5b1` 预订工作台/经营看板、`M5b2` 团购/订单/退款审核、`M5b3` 门店完整草稿审核与 `M5b4` 商户点评回复/申诉,不等于本文档里的完整目标态库表。
- 当前已直接落库的核心表除浏览、认证、点评、交易和预订表外，还包括 `merchant_operator`、`merchant_role`、`merchant_operator_role`、`merchant_operator_shop`、`merchant_application`、`merchant_operation_log`、`merchant_shop_change`、`merchant_shop_change_photo`、`merchant_shop_change_dish`、`review_merchant_reply`、`merchant_review_appeal` 与 `shop_view_daily`。其中本文后续的 `merchant_staff` / `merchant_permission` 等是较早目标态草案，当前运行代码和导库必须以 `merchant_operator` 模型及 `sql/mysql/01_schema.sql` 为准。
- `02_seed_data.sql` 已预置浏览数据、公开点评、点评图片、点赞/评论演示数据、点评审核演示任务和两个可直接密码登录的 C 端演示账号,方便你导库后直接对照当前已完成能力。
- 其中 `review` / `review_image` / `review_like` / `review_comment` / `review_report` / `review_merchant_reply` / `merchant_review_appeal` / `audit_task` / `audit_log` 已支撑当前代码的点评发布、互动、审核、商家回复、商户申诉和门店评分聚合闭环。
- 账号绑定 / 改密码 / 公开用户主页直接复用 `app_user`、`verification_code`、`review` 等现有表;登录用户搜索历史使用 `search_history`。
- `POST /auth/send-code` 验证码限流和 `Idempotency-Key` 重复提交保护当前已在代码里落地,默认本地内存,配置 `APP_STATE_STORE_PROVIDER=redis` 后可切 Redis,不依赖额外 MySQL 表;本文后面的 `idempotency_record` / `rate_limit_log` 仍是审计增强的目标态设计。
- 管理端初始账号、内置角色、权限关联与 `CN/EU` 区域范围均由 `02_seed_data.sql` 写入 `admin_user`、`admin_role`、`admin_permission`、`admin_user_role`、`admin_role_permission` 和 `admin_region_scope`；密码只以 BCrypt `password_hash` 存储。
- 例如当前运行代码使用的是 `app_user` 表,而本文档为了表达全量目标态,仍保留 `user` 等更完整命名。**真正导库时先以 `sql/mysql` 为准**,别拿目标态文档硬怼现代码。
- 当前代码的点评举报表实际名称是 `review_report`;本文后面若提到全站统一 `report` 表,那是帖子/私信举报一起落地时的目标态预案,别拿它指导当前导库。

**主键与通用字段**
- 主键:`id BIGINT UNSIGNED AUTO_INCREMENT`。
- 每张业务表默认带:
  - `created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP`
  - `updated_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP`
  - `is_deleted TINYINT NOT NULL DEFAULT 0` —— 逻辑删除(0 正常 / 1 已删),不物理删数据。

**区域隔离(§0 核心原则)**
- 凡是"分国内/海外"的业务数据,统一带 `region VARCHAR(8)`,取值 `CN` / `EU`(后续可扩 `US`/`AU` 等)。
- `X-Region` 对应 `request_region`;当前 `Web` 默认 `CN` 但已支持切换 `CN / EU`,`APP` 规划默认 `EU`。
- 涉及 `data_region` 的表:`shop`、`city`、`area`、`category`(可共享也可分区,见下)、`deal`、`review`、`post`、`merchant` 等。
- `user.preferred_region` 只表达账号默认偏好,不用于数据隔离。
- 查询默认按 `data_region` 过滤。

**金额与货币**
- 金额一律用 `DECIMAL(10,2)`(分位精确),**不用 float**。
- 带货币的表加 `currency CHAR(3)`(`CNY` / `EUR` / `GBP`),ISO 4217。

**状态字段**
- 用 `TINYINT` + 代码注释枚举,不用中文字符串存状态。

**外键**
- 逻辑外键(靠字段 + 索引关联),**不建物理 FK 约束**(便于分库分表、批量导入、性能),关系在应用层保证。

**索引命名**
- 主键 `PRIMARY`;唯一 `uk_字段`;普通 `idx_字段`。

**大文本 / JSON**
- 富文本、规则、扩展字段用 `TEXT` 或 `JSON`(MySQL 5.7+ 支持 JSON 类型)。

## 1. 用户模块

### 1.1 `user` 用户主表
```sql
CREATE TABLE `user` (
  `id`            BIGINT UNSIGNED NOT NULL AUTO_INCREMENT,
  `nickname`      VARCHAR(64)     NOT NULL DEFAULT '' COMMENT '昵称',
  `avatar`        VARCHAR(255)    NOT NULL DEFAULT '' COMMENT '头像URL',
  `email`         VARCHAR(128)    DEFAULT NULL COMMENT '邮箱(登录方式一)',
  `phone`         VARCHAR(32)     DEFAULT NULL COMMENT '手机号,含国际区号,如 +86/+44(登录方式二)',
  `password_hash` VARCHAR(100)    DEFAULT NULL COMMENT '密码哈希(BCrypt),验证码登录可为空',
  `gender`        TINYINT         NOT NULL DEFAULT 0 COMMENT '0未知 1男 2女',
  `signature`     VARCHAR(255)    NOT NULL DEFAULT '' COMMENT '个性签名',
  `preferred_region` VARCHAR(8)   NOT NULL DEFAULT 'CN' COMMENT '默认浏览偏好 CN/EU,不参与数据隔离',
  `growth_value`  INT UNSIGNED    NOT NULL DEFAULT 0 COMMENT '成长值',
  `level`         TINYINT         NOT NULL DEFAULT 1 COMMENT '用户等级 Lv1-Lv8',
  `points`        INT UNSIGNED    NOT NULL DEFAULT 0 COMMENT '积分(可用)',
  `status`        TINYINT         NOT NULL DEFAULT 1 COMMENT '1正常 2封禁 3注销',
  `last_login_at` DATETIME        DEFAULT NULL COMMENT '最后登录时间',
  `created_at`    DATETIME        NOT NULL DEFAULT CURRENT_TIMESTAMP,
  `updated_at`    DATETIME        NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  `is_deleted`    TINYINT         NOT NULL DEFAULT 0,
  PRIMARY KEY (`id`),
  UNIQUE KEY `uk_email` (`email`),
  UNIQUE KEY `uk_phone` (`phone`),
  KEY `idx_preferred_region` (`preferred_region`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='用户表';
```
> 说明:邮箱、手机号均可为空但各自唯一(允许只绑一种,后续可绑另一种)。注册即登录 → 首次验证通过时插入一条记录。`preferred_region` 只做账号偏好,不影响 PC/APP 的默认分区规则。

### 2.7.1 `merchant_shop_change*` 门店完整候选快照

当前运行代码使用三张表承载 M5b3，不直接在待审阶段修改线上 `shop/shop_photo/dish`：

- `merchant_shop_change`：保存商户、操作者、区域、`change_type(1新建 2修改)`、目标门店、`base_updated_at`、全部商户可编辑字段及 `status(0草稿 1待审 2通过 3驳回 4失效)`。
- `merchant_shop_change_photo`：保存候选照片的 `image_url/photo_type/sort_no`，每次保存按完整相册先删后插。
- `merchant_shop_change_dish`：保存候选菜品的 `name/price/recommend_reason/sort_no`，每次保存按完整菜单先删后插。

关键索引为 `idx_shop_change_merchant_status(merchant_id,region,status)` 与 `idx_shop_change_target_status(target_shop_id,status)`。修改门店审核时用 `base_updated_at` 与锁定后的线上 `shop.updated_at` 比较；通过后只覆盖商户可编辑字段，评分、点评数、`has_deal`、`merchant_id` 和 `region` 不进入更新 SQL。

## 8. 后台管理模块(RBAC + 审核 + 审计)

### 8.1 已落地的管理员数据库 RBAC

当前运行代码和 MySQL/H2 初始化脚本使用下列真实表名，不使用旧的 `admin`、`role`、`permission` 或 `role_permission` 目标态名称：

| 表 | 关键字段 / 职责 |
|---|---|
| `admin_user` | `id/account/password_hash/name/status/last_login_at`；管理员账号，密码为 BCrypt hash，`status=1` 启用、`2` 停用 |
| `admin_role` | `id/code/name/description/status/built_in`；角色定义，内置角色受后端保护 |
| `admin_permission` | `id/code/name/category/permission_type/status`；菜单/动作/API 权限点，如 `system:admin:write`、`audit:review:read` |
| `admin_user_role` | `(admin_id, role_id)` 复合主键；管理员与角色多对多关联 |
| `admin_role_permission` | `(role_id, permission_id)` 复合主键；角色与权限点多对多关联 |
| `admin_region_scope` | `(admin_id, region)` 复合主键；管理员可访问的 `CN/EU` 区域范围 |

权限计算链路为 `admin_user → admin_user_role → admin_role → admin_role_permission → admin_permission`；区域范围独立从 `admin_region_scope` 加载。token 只保存管理员标识与过期时间，受保护请求会重新读取启用账号、角色、权限和区域范围。角色停用后，旧 token 仍可取得 `200` 的 `auth/me`，但权限被重新计算并收回，固定受限 API 返回 `403`，动态审核列表可返回 `200` 空结果；管理员账号停用后，旧 token 才会在下一次请求返回 `401`。

种子数据内置 `super_admin`、`content_auditor`、`merchant_auditor`、`operations_manager` 与 `data_operator` 五种角色；`super_admin` 拥有全部当前权限点，初始 `admin` 账号覆盖 `CN/EU`。这只覆盖当前管理员身份与权限基础，不代表分类/城市/商圈、用户治理、订单退款对账、运营活动或审计/隐私任务查询已经完成。

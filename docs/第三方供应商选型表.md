# 大众点评(仿)项目 · 第三方供应商选型表

> 版本:v0.1
> 最后更新:2026-07-09
> 调研基准:基于 2026-07-09 前后官方文档的能力说明整理,**不固化价格**
> 配套文档:`第三方集成与合规.md`、`系统架构设计.md`、`接口设计.md`、`M1-M2实施计划与验收清单.md`、`测试清单与验收用例.md`、`上线准备清单.md`、`执行版联系人与台账模板.md`

## 1. 文档目标

这份文档只解决一件事:把“邮件、短信、对象存储、CDN、内容审核、推送”这些外部能力的供应商选型先拍成**一期可落地版本**。

不在这里干的事:

- 不写精确价格表。价格这玩意变得比天气还快,采购前再核。
- 不做无穷无尽的厂商横评。我们是做项目,不是写咨询报告。
- 不追求一期就双活多供应商热备。先把适配层留好,别把自己搞进复杂度泥潭。

## 2. 选型原则

- **区域优先**:国内区优先国内链路、合规和可达性;欧洲区优先全球可达、EU 数据处理和海外生态。
- **少厂商优先**:能同云栈闭环就别东拼西凑,减少账号、密钥、计费、故障面。
- **适配层隔离**:业务层不直接绑死 SDK,统一经 `mail/sms/storage/cdn/moderation/push adapter`。
- **一期够用**:先满足验证码、通知邮件、图片上传、静态分发、点评/帖子审核,别上来就追求满配。
- **可替换**:配置项、Webhook、错误码、回调都要抽象,后续切供应商不至于把业务代码拆烂。

## 3. 一期推荐总方案

| 能力 | CN 区推荐 | EU 区推荐 | 一期结论 |
|---|---|---|---|
| 邮件 | 阿里云 DirectMail | Amazon SES | 通过适配层双实现,`M2` 先接主方案 |
| 短信 | 阿里云 SMS | Twilio Messaging(IE1 区域) | 验证码与通知分模板管理 |
| 对象存储 | 阿里云 OSS | Amazon S3 | 上传统一走文件服务签名 |
| CDN | 阿里云 CDN | Amazon CloudFront | 静态资源、图片分发走 CDN |
| 内容审核 | 阿里云 AI Guardrails(内容安全) | Amazon Rekognition + 文本规则引擎 | EU 文本先走规则库 + 人审 |
| 移动推送 | FCM / APNs | FCM / APNs | 这是平台依赖,不是自由选型项 |

## 4. 最终建议组合

### 4.1 CN 区

建议一期直接走**阿里云全家桶**:

- 邮件:DirectMail
- 短信:SMS
- 对象存储:OSS
- CDN:CDN
- 内容审核:AI Guardrails

好处:

- 控制台、AK/SK、RAM 权限、告警和账单都在一个体系里。
- 国内链路和实名、模板审核这些破事统一在一朵云里处理,省很多沟通成本。
- `M2` 登录、`M3+` 图片/社区审核都能顺滑接上。

### 4.2 EU 区

建议一期走**AWS + Twilio + FCM/APNs**:

- 邮件:Amazon SES
- 短信:Twilio Messaging(IE1)
- 对象存储:Amazon S3
- CDN:Amazon CloudFront
- 内容审核:Amazon Rekognition
- 推送:FCM / APNs

好处:

- 海外交付链路成熟,和 APP 欧洲首发定位匹配。
- `S3 + CloudFront + Rekognition` 比较顺手,图片上传、分发、审核一条线好接。
- Twilio 的短信生态更成熟,对欧洲手机号、区域凭证、运营商路由更省心。

### 4.3 不建议的一期做法

- 不建议 CN/EU 共用一套单厂商发信/发短信,后面要么可达性差,要么合规麻烦。
- 不建议一开始就做双供应商热切换,那是有钱有人之后再玩的。
- 不建议先接一堆“便宜但陌生”的聚合厂商,账单看着省,排障能把人折腾裂开。

## 5. 分能力选型说明

### 5.1 邮件

| 项 | CN 区 | EU 区 |
|---|---|---|
| 主方案 | 阿里云 DirectMail | Amazon SES |
| 主要用途 | 注册、登录、找回密码、系统通知 | 注册、登录、找回密码、系统通知 |
| 接入方式 | API / SMTP | API / SMTP |
| 一期建议 | 只做事务型邮件 | 只做事务型邮件 |

接入要求:

- 发信域名、发信地址、回信地址都要提前验证。
- 必配 `SPF / DKIM / DMARC`。
- 账号预热单独做,别第一天就猛灌大量邮件。
- 邮件模板要区分 `login/register/reset/delete/system_notice`。

风险提醒:

- `M2` 只需要事务型邮件,营销邮件别提前开,那是给自己加 deliverability 的麻烦。
- 海外邮件回执、退信、投诉事件最好统一落回调日志,方便封禁无效地址。

### 5.2 短信

| 项 | CN 区 | EU 区 |
|---|---|---|
| 主方案 | 阿里云 SMS | Twilio Messaging(IE1) |
| 主要用途 | 验证码、系统通知 | 验证码、系统通知 |
| 接入方式 | API / SDK | REST API |
| 一期建议 | 先只开验证码与重要通知 | 先只开验证码与重要通知 |

接入要求:

- 模板、签名、资质、委托授权材料要提前准备。
- 短信发送统一走验证码中心,不允许业务方乱发。
- 限流规则以 `接口设计.md` 为准,别让短信接口变提款机。
- EU 区使用区域化凭证和专用配置,不要和别的区域混着用。

风险提醒:

- 国际短信有国家覆盖、模板审核、号码合规等差异,别把国内经验生搬硬套过去。
- 短信是最容易烧钱的能力之一,风控和报警要比“能发出来”更早做。

### 5.3 对象存储

| 项 | CN 区 | EU 区 |
|---|---|---|
| 主方案 | 阿里云 OSS | Amazon S3 |
| 存储对象 | 商户图、点评图、帖子图、导入文件、隐私导出包 | 同左 |
| 上传方式 | 服务端签名 + 客户端直传 | 服务端签名 + 客户端直传 |
| 一期建议 | 公私桶分离,按区域分桶 | 公私桶分离,按区域分桶 |

接入要求:

- Bucket 按 `region` 分开。
- 上传白名单限制文件类型、尺寸、数量。
- 业务侧永远存对象 key/标准 URL,别硬拼临时域名。
- 隐私导出包和用户上传图不要放同一生命周期策略。

风险提醒:

- 公有读图和私有导出包的权限边界要分干净。
- 跨区回源、跨区存储看着像省事,实际上容易把合规和延迟搞成一坨。

### 5.4 CDN

| 项 | CN 区 | EU 区 |
|---|---|---|
| 主方案 | 阿里云 CDN | Amazon CloudFront |
| 回源对象 | OSS | S3 |
| 一期用途 | 商户图、点评图、静态资源 | 商户图、点评图、静态资源 |
| 核心配置 | HTTPS、缓存 TTL、CNAME、日志 | HTTPS、缓存策略、行为规则、日志 |

接入要求:

- 图片、静态资源、上传回源策略分开配。
- 缓存刷新和版本化发布要定规矩,别全靠人工点按钮。
- 强制 HTTPS。
- 统一接访问日志或监控,为后面排障留证据。

风险提醒:

- CDN 配不好,最典型的结果就是“功能没错,用户全看到旧图”。
- 一期不追求复杂边缘逻辑,先把缓存、回源、证书、监控配稳定。

### 5.5 内容审核

| 项 | CN 区 | EU 区 |
|---|---|---|
| 主方案 | 阿里云 AI Guardrails(内容安全) | Amazon Rekognition + 文本规则引擎 |
| 审核对象 | 点评文本、点评图、帖子图文、私信举报复审 | 点评图、帖子图、视频留口子;文本先规则库 + 人审 |
| 调用方式 | API / 异步回调 | API / 异步结果 |
| 一期建议 | 机审 + 人审闭环 | 机审(图片/视频) + 文本规则 + 人审 |

说明:

- CN 区直接接成熟内容审核能力最省事。
- EU 区如果只靠 Rekognition,图像/视频能兜住,**纯文本语义审核不够完整**,一期建议配合自定义敏感词库、规则引擎和人审。

风险提醒:

- 内容审核不能只做图片不做文本,否则社区和点评迟早出事。
- 审核误杀、漏判都要留复核入口,别把机审当法官。

### 5.6 推送

这一项其实没啥好装神秘的:

- Android 走 `FCM`
- iOS 走 `APNs`
- PC 在线实时通知走项目自己的 `WebSocket`

一期重点不是“挑推送厂商”,而是把 `user_device`、token 生命周期、失败停用和消息回补做好。

## 6. 适配层与配置项建议

### 6.1 适配层接口

建议按能力抽 6 个适配器:

- `MailProvider`
- `SmsProvider`
- `StorageProvider`
- `CdnProvider`
- `ModerationProvider`
- `PushProvider`

统一返回:

- `success`
- `provider`
- `requestId`
- `rawCode`
- `rawMessage`
- `retryable`

### 6.2 环境变量建议

#### CN

- `DIRECTMAIL_*`
- `ALIYUN_SMS_*`
- `OSS_*`
- `ALIYUN_CDN_*`
- `ALIYUN_MODERATION_*`

#### EU

- `SES_*`
- `TWILIO_*`
- `S3_*`
- `CLOUDFRONT_*`
- `REKOGNITION_*`
- `FCM_*`
- `APNS_*`

## 7. 按里程碑接入顺序

### 7.1 M1

- 对象存储测试环境
- CDN 测试环境
- 种子导入文件存储

### 7.2 M2

- 邮件验证码
- 短信验证码
- 点评图片上传

### 7.3 M3-M5

- 内容审核正式接入
- 推送与 WebSocket 深化

## 8. 采购 / 上线前检查清单

- [ ] 主域名、子域名、发信域名归属明确
- [ ] 邮件域名验证、DKIM、SPF、DMARC 配好
- [ ] 短信签名、模板、资质审核完成
- [ ] 存储桶按区域建好,权限策略过一遍
- [ ] CDN 域名、证书、缓存策略、日志策略过一遍
- [ ] 审核服务压测一轮,评估误杀率和回调延迟
- [ ] 各供应商 AK/SK 或等价凭证全部进入密钥管理
- [ ] 告警、重试、降级和审计日志方案补齐

## 9. 一期最终拍板建议

如果现在就要往前走,我的建议很直接:

- **CN 区**:全阿里云。
- **EU 区**:`SES + Twilio + S3 + CloudFront + Rekognition + FCM/APNs`。
- **技术实现**:统一适配层 + 环境变量切换。
- **运维策略**:一期先单主供应商,预留冷切备选,不做双活。

这套组合的好处不是“最炫”,而是**最像能活着上线的方案**。

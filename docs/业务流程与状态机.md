# 大众点评(仿)项目 · 业务流程与状态机

> 版本:v0.6(补齐本地达人认证状态机)
> 最后更新:2026-07-21
> 配套文档:`需求文档.md`、`接口设计.md`、`数据库设计.md`、`权限矩阵.md`、`文档差异与待确认清单.md`

## 1. 文档作用

需求讲“要做什么”,接口和库表讲“长什么样”,但真正干活时最容易翻车的是“怎么流转”。本文把核心业务流程和状态机补上,避免后面每个人脑子里都有一版自己的剧情。

### 1.1 区域语义

- `request_region`: 当前请求上下文区域,由客户端请求头带入。当前 `Web` 默认 `CN` 但已支持切换 `CN / EU`,`APP` 规划默认 `EU`。
- `data_region`: 业务数据所属区域,例如门店、团购、点评、帖子。
- `preferred_region`: 用户默认浏览偏好,仅用于后续偏好和消息策略,不参与访问控制。
- `management_scope`: 商户主账号或平台管理员可操作的数据区域范围。

## 2. 登录与游客拦截

### 2.1 游客态规则

- 首页、频道、列表、详情、搜索、榜单、公开点评、公开社区内容允许游客访问。
- 写点评、点赞、评论、收藏、下单、私信、查看我的页、消息通知需要登录。

### 2.2 行为拦截流程

1. 用户在游客态触发受限动作。
2. 前端记录 `sourceAction` 与必要参数。
3. 弹出登录浮层,不强制整页跳转。
4. 登录成功后恢复原动作。
5. 若恢复失败,给出明确错误,不能让用户以为点了个寂寞。

### 2.3 登录风控建议

- 发验证码限流:账号维度、IP 维度、设备维度三层限流。
- refresh token 支持主动失效。
- 邮箱/手机号换绑要二次校验。

### 2.4 收藏流程

一期收藏只保留两类对象:

- `1店铺`
- `2帖子`

流程建议:

1. 登录用户在店铺详情或帖子详情触发收藏。
2. 系统按 `user_id + target_type + target_id` 幂等去重。
3. 收藏成功后更新收藏列表与对应计数。
4. 取消收藏走同一资源反向操作,不保留“软删除后再恢复”这套花活。

## 3. 点评发布与审核

### 3.1 流程

1. 用户上传图片。
2. 用户提交点评与评分。
3. 系统写入点评主表与媒体表,状态进入待审。
4. 生成审核任务。
5. 审核通过后对外可见,并回写商户聚合评分。
6. 审核驳回后,用户可见驳回原因并选择修改后重提。

### 3.2 状态建议

| 对象 | 字段 | 状态 | 说明 |
|---|---|---|---|
| review | `audit_status` | `0待审 1通过 2驳回` | 当前数据库已定义 |
| review | `status` | `1正常 2隐藏` | 运营下架、违规隐藏 |
| user_expert_certification | `status` | `0未申请 1待审核 2已通过 3已驳回` | 本地达人认证申请，按 `user_id + region` 唯一 |
| audit_task | `status` | `0待人审 1通过 2驳回` | 审核中心统一口径 |

### 3.3 必补动作

- 点评编辑后重新进入审核。
- 举报后可再次触发复审。
- 点评删除是否物理删,当前建议仍走逻辑删除。

## 4. 团购订单、支付、券码、退款

### 4.1 下单支付主流程

1. 用户选择团购并提交订单。
2. 系统校验库存、限购、有效期、区域、门店状态。
3. 创建待支付订单并锁库存。
4. 发起第三方支付。
5. 支付成功后生成券码。
6. 用户可查看“我的订单”和“我的券”。

### 4.2 订单状态拆分

当前数据库把 `order.pay_status` 和 `order.status` 拆开是对的,别图省事合并成一坨。

| 字段 | 建议状态 | 说明 |
|---|---|---|
| `order.pay_status` | `0待支付 1已支付 2已退款 3部分退款` | 表达资金状态 |
| `order.status` | `1正常 2已关闭` | 表达订单是否还有效 |
| `coupon.status` | `1待使用 2已使用 3已过期 4已退款` | 表达核销侧状态 |
| `refund.status` | `0申请中 1退款成功 2驳回` | 表达退款处理状态 |

### 4.3 支付补偿

- 前端同步支付成功不代表最终成功,以后端异步回调为准。
- 回调必须幂等。
- 回调超时或异常时,通过对账任务补偿。
- 券码生成必须和支付成功保持一致,避免“钱扣了券没了”的经典事故。

### 4.4 退款流程

1. 用户申请退款。
2. 系统校验订单已支付、未存在已核销券、未存在历史退款申请，写入 `refund.status=0`；订单继续保持已支付。
3. 商户员工按 `order:refund` 权限、商户归属、区域和门店范围审核。
4. 驳回时仅更新退款状态与审核信息，订单和券保持原状态。
5. 通过时在同一事务内更新退款、订单、待使用券并恢复团购库存/销量；当前模拟渠道不调用真实支付网关退款 SDK。
6. 真实原路退款、部分退款和自动退款规则继续归 M6/支付集成阶段。

## 5. 预订 / 订座

### 5.1 设计原则

- 预订与团购订单共用商户工作台,但状态机独立。
- 时段容量、确认方式、提醒、改期必须可追踪。
- 一期不做复杂排队与候补,把确认、拒绝、改期、到店、爽约这几个核心动作先跑通。

### 5.2 建议流程

1. 用户先查看门店可订日期/时段与剩余容量。
2. 用户选择门店、日期、时段、人数并提交预订。
3. 系统校验门店营业状态、时段容量、人数上限、取消截止时间。
4. 如果门店时段配置为自动确认,直接进入“已确认”;否则进入“待确认”。
5. 商户在 SLA 内确认、拒绝,或与用户沟通后执行改期。
6. 到店前由系统发送提醒。
7. 到店后由店员或店长确认“已到店”。
8. 超过宽限时间仍未到店,系统或店员可标记爽约。

### 5.3 建议状态

当前采用下面这套状态,不再把“用户取消”和“商户拒绝”混成一锅。

| 建议状态 | 含义 | 备注 |
|---|---|---|
| `0待确认` | 刚提交,等商户处理 | 人工确认门店默认入口 |
| `1已确认` | 自动确认或商户确认接单 | 可继续改期 |
| `2已到店` | 店员/店长确认用户到店 | 终态 |
| `3用户取消` | 用户在允许时间内主动取消 | 终态 |
| `4商户拒绝` | 商户拒绝或门店无法接待 | 终态 |
| `5爽约` | 到店时间过后仍未到店 | 终态 |

### 5.4 补充表建议

- `reservation_slot`: 门店按日期/时段配置容量、自动确认规则。
- `reservation_change_log`: 记录创建、确认、拒绝、取消、改期、提醒、到店、爽约动作。

### 5.5 改期规则

- 用户和商户都可发起改期。
- 改期成功后:
  - 自动确认门店:直接更新到新时段,状态保持 `1已确认`。
  - 人工确认门店:更新到新时段后回到 `0待确认`。
- 改期失败不改变原状态,但必须写变更日志。

### 5.6 关键时效

- 商户确认 SLA:按门店配置,默认 30 分钟。
- 用户取消截止时间:按门店配置,默认到店前 2 小时。
- 到店提醒:默认到店前 2 小时与 30 分钟各一次。
- 爽约宽限时间:默认到店时间后 30 分钟。

## 6. 商户入驻与审核

### 6.1 流程

1. 商户注册账号。
2. 提交资质与主体信息。
3. 平台生成审核任务。
4. 审核通过后才允许创建/提交门店与团购。
5. 审核驳回后可重提。

### 6.2 审核对象

- 商户主体资质。
- 门店信息。
- 团购/代金券。
- 点评/帖子内容。

### 6.3 商户账号与员工授权

1. 商户主账号创建员工账号。
2. 主账号为员工绑定角色与门店范围。
3. 店长可处理门店编辑、团购、预订、点评回复。
4. 核销员只允许做券码核销与到店确认。
5. 客服运营只允许做点评回复、申诉、消息处理。
6. 角色调整必须留审计记录,不能谁想改就改。

### 6.4 门店基础资料 / 相册 / 菜单完整草稿审核

1. 商户新建门店时创建空草稿；修改门店时从当前线上 `shop/shop_photo/dish` 复制完整候选快照，并记录 `base_updated_at`。
2. 基础资料、相册、菜单分别保存，但都落在同一个 `merchant_shop_change` 申请下；保存照片和菜品采用完整快照替换，不做半拉增量状态。
3. 提交前校验区域币种、分类/城市/商圈、经纬度、价格、照片/菜单数量，以及封面必须存在于门店图相册中。
4. 提交后草稿 `0 -> 1`，生成 `audit_task.biz_type=5,status=0`；此时公开查询仍读旧线上数据。
5. 管理员通过新门店申请时，在单事务内创建 `shop` 并写入候选照片/菜单；通过修改申请时锁定线上门店，比较 `shop.updated_at` 与 `base_updated_at`，一致才整体替换商户可编辑字段、照片和菜单。
6. 修改门店不会覆盖评分、分项评分、点评数、`has_deal`、商户归属或区域；事务提交后发布搜索索引变更事件。
7. 管理员驳回时审核任务 `0 -> 2`、草稿 `1 -> 3`，记录原因和审核人，不修改线上三张表，也不发布索引事件。
8. 被驳回草稿首次保存时 `3 -> 0` 并清空旧审核字段；重新提交会生成新的待审核任务。重复提交、重复审核和跨区域审核都必须拦截。

状态表:

| 对象 | 状态 | 允许动作 |
|---|---|---|
| 门店草稿 | `0草稿` | 保存基础资料/照片/菜单、提交 |
| 门店草稿 | `1待审核` | 仅查看，等待管理员通过或驳回 |
| 门店草稿 | `2已通过` | 终态，线上快照已应用 |
| 门店草稿 | `3已驳回` | 查看原因，修改后自动回草稿并可重提 |
| 门店草稿 | `4已失效` | 终态，不允许继续编辑 |

### 6.5 商户点评回复与申诉

商户点评经营只处理已公开、未删除、归属当前商户和当前区域的点评；跨商户、跨区域、非授权门店或非公开点评都按资源不存在处理。

回复流程：

1. 商户员工按 `review:reply` 权限和门店范围查看公开点评。
2. 首次回复写入 `review_merchant_reply`，再次回复覆盖当前内容。
3. C 端公开点评列表和详情返回可空 `merchantReply`。
4. 点评被隐藏或删除后，回复记录仍保留，但不再单独公开暴露。

申诉流程：

1. 商户员工按 `review:appeal` 权限创建或复用申诉草稿。
2. 保存理由和证据后提交，申诉 `0草稿 -> 1待审核`，生成 `audit_task.biz_type=6,status=0`。
3. 管理员通过申诉时，审核任务 `0 -> 1`，申诉 `1 -> 2`，点评 `audit_status -> 2`，并重算门店评分和点评数。
4. 管理员驳回申诉时，审核任务 `0 -> 2`，申诉 `1 -> 3`，点评继续公开。
5. 驳回后商户首次保存会把申诉 `3 -> 0` 并清空旧审核字段，随后可重新提交。
6. 用户编辑或删除点评时，旧申诉和对应待审任务统一失效，避免拿旧正文继续审核。

状态表：

| 对象 | 状态 | 说明 |
|---|---|---|
| 商户点评申诉 | `0草稿` | 可编辑理由和证据，可提交 |
| 商户点评申诉 | `1待审核` | 不允许编辑，等待管理端审核 |
| 商户点评申诉 | `2已通过` | 终态，本轮申诉已隐藏点评 |
| 商户点评申诉 | `3已驳回` | 可修改后重提 |
| 商户点评申诉 | `4已失效` | 点评编辑/删除或版本变化后旧申诉不可继续处理 |

### 6.6 本地达人认证

1. 登录用户在资料页填写申请理由并提交达人认证。
2. 系统按 `user_id + region` 查询当前申请：若已有待审记录或已通过且仍有效，直接拦截重复提交。
3. 首次提交时写入 `user_expert_certification.status=1`；驳回后重提时复用原记录、清空旧审核字段，并把旧待审 `audit_task` 置失效。
4. 每次有效提交都会生成 `audit_task.biz_type=7,status=0`，统一进入管理端审核中心。
5. 管理员在 `/audit/expert-certifications` 处理任务：通过时认证 `1 -> 2`，写入 `audited_at/effective_start_at`；驳回时认证 `1 -> 3`，写入 `reject_reason/audited_at`。
6. 只有 `status=2` 且当前时间命中有效期时，公开用户主页、点评和帖子作者信息才展示 `code=local_expert,label=本地达人`；待审、驳回和失效都不公开挂标。

状态表：

| 对象 | 状态 | 允许动作 |
|---|---|---|
| 本地达人认证 | `0未申请` | 提交申请 |
| 本地达人认证 | `1待审核` | 仅查看状态，等待管理端处理 |
| 本地达人认证 | `2已通过` | 终态，公开资料与作者信息挂标 |
| 本地达人认证 | `3已驳回` | 查看原因，修改理由后可重提 |

## 7. 社区、举报、内容治理

### 7.1 帖子流程

1. 用户发帖。
2. 帖子进入审核。
3. 审核通过后进入推荐流/关注流。
4. 被举报后可二次复审或下架。

### 7.2 举报统一口径

当前 `report.target_type` 只覆盖点评/评论/帖子/用户,和“私信先只做举报”对不上。建议把举报对象统一扩成:

- 点评
- 点评评论
- 帖子
- 用户
- 私信消息
- 私信会话

举报流程建议:

1. 用户发起举报并选择对象类型、原因、补充说明。
2. 系统落举报记录并通知审核中心。
3. 内容审核员处理后回写状态与结论。
4. 命中高风险规则时,可先临时隐藏内容再进入复审。

### 7.3 社区通知

- 被点赞、被评论、被关注、被私信都应落统一通知中心。
- `@提醒` 建议单列消息类型,别混在普通评论里装死。

## 8. 通知与推送

### 8.1 发送路径

- PC 在线:WebSocket 推送。
- APP 在线:长连接或前台消息。
- APP 离线:FCM / APNs。
- 任何推送失败都不能丢持久化记录。

### 8.2 当前要求

- 设备注册流程。
- push token 更新与失效清理。
- 通知已读、批量已读、聚合去重策略。

### 8.3 设备注册与 token 生命周期

1. 用户登录 APP 后注册设备信息与 push token。
2. 用户换机、重装、token 刷新时更新同一设备记录。
3. 登出时可主动解绑 token,但保留设备记录用于审计。
4. 推送失败达到阈值后自动停用该 token。
5. 用户注销时清理有效 push token 并停止消息投递。

## 9. 运营配置发布

### 9.1 榜单规则发布

1. 运营在后台编辑 `rank_config` 草稿。
2. 保存后生成新版本,但不立即影响线上榜单。
3. 点击发布后,系统触发重算任务生成新的 `rank` / `rank_item` 快照。
4. 新快照生成成功后切流并清理 Redis 榜单缓存。
5. 若重算失败,保留旧版本在线,别让首页榜单直接黑屏。

### 9.2 运营活动上下线

1. 运营创建活动主体并配置资源项。
2. 活动状态从 `草稿 -> 待上线 -> 上线中 -> 已下线/已结束` 流转。
3. 到 `startAt` 由定时任务自动上线,到 `endAt` 自动下线。
4. 资源项排序、角标、跳转参数都算活动版本的一部分,变更后要同步刷新缓存。

## 10. 隐私中心

### 10.1 数据导出

1. 用户在设置页发起导出申请。
2. 系统校验登录态、频控和导出范围。
3. 创建 `privacy_export_task` 并异步聚合账号、订单、点评、帖子、私信等数据。
4. 打包完成后生成短时下载链接并通知用户。
5. 下载有效期过后,文件自动清理,任务状态切为过期。

建议状态:

| 对象 | 字段 | 状态 | 说明 |
|---|---|---|---|
| `privacy_export_task` | `status` | `0待处理 1处理中 2可下载 3已过期 4失败 5已取消` | 导出任务全流程 |

### 10.2 数据删除 / 注销

1. 用户发起删除申请。
2. 系统要求密码或验证码二次确认;验证码必须走 `scene=delete`。
3. 创建 `privacy_delete_task`,进入冷静期。
4. 冷静期内用户可撤销;到期后系统执行 token 失效、设备停用、数据匿名化。
5. 订单、支付、审计等必须保留的记录做脱敏保留,不是一股脑物理删库。

建议状态:

| 对象 | 字段 | 状态 | 说明 |
|---|---|---|---|
| `privacy_delete_task` | `status` | `0待确认 1冷静期中 2处理中 3已完成 4已取消 5已驳回` | 删除/注销任务 |

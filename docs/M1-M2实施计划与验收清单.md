## 2. 里程碑范围

### 2.1 M1 目标

对应 `需求文档.md §7` 的 `M1`:

- 区域架构打底(`request_region` / `data_region`)
- 后端基础:用户、商户、分类、地理
- 运营录种子商户通道
- PC 端免登录浏览:首页 / 列表 / 详情

### 2.2 M2 目标

对应 `需求文档.md §7` 的 `M2`:

- 登录鉴权(邮箱 + 手机号 + 行为拦截)
- 点评体系(发点评、评分聚合、点评列表)
- 基础用户中心(我的资料、我的点评)

### 2.3 明确不在 M1/M2 的范围

下面这些别硬塞进来,塞进来就不是 `M1/M2` 了:

- 团购、下单、支付、退款
- 预订 / 订座
- B 端商户工作台完整能力
- 管理端完整 RBAC 和运营活动
- APP、Flutter、社区/社交
- WebSocket 实时通知
- Elasticsearch 搜索

## 3. 实施前提

- 后端采用单体式模块化,先不拆微服务。
- 数据库一期按当前 `数据库设计.md` 的主表落库,不做分库分表。
- `M1` 列表/详情先走 MySQL,`M3` 再上 ES。
- 对象存储和 CDN 可以先接测试环境,但图片链路在 `M2` 前必须跑通,因为点评要传图。
- 审核中心 `M2` 至少要有**最小可用版本**,不然点评发出来没地方审。

## 4. 建议节奏

默认按 `8 周` 拆,每 `2 周` 一个迭代:

| 周期 | 里程碑 | 目标 |
|---|---|---|
| 第 1-2 周 | Sprint 1 | 后端骨架、区域/地理/商户主数据、种子导入打底 |
| 第 3-4 周 | Sprint 2 | PC 浏览链路闭环,M1 验收 |
| 第 5-6 周 | Sprint 3 | 认证鉴权、用户中心、验证码链路 |
| 第 7-8 周 | Sprint 4 | 点评发布、审核、聚合评分,M2 验收 |

如果人手更少,可以拉成 `10-12 周`;如果人多,可以压缩工期,但别把测试和数据初始化砍了。

## 7. 依赖关系

### 7.1 M1 关键依赖

- 基础数据准备在前,否则列表页就是空壳。
- 导入通道在前,否则运营没法造数据。
- 区域上下文在前,否则后面所有接口都得返工。

### 7.2 M2 关键依赖

- 验证码服务供应商在 `M2` 开始前定掉。
- 对象存储在 `M2` 开始前跑通,否则点评图片没法验收。
- 审核最小后台必须同步开发,不能让点评上线后靠数据库手改。

## 8. 风险与应对

| 风险 | 影响 | 应对 |
|---|---|---|
| 区域字段语义被写乱 | 后续接口和库表大面积返工 | 所有接口和 SQL 评审都强校验 `request_region/data_region` |
| 种子数据质量差 | M1 看起来“功能做了”,其实没法演示 | 提前准备模板、失败回灌和运营验收 |
| 验证码供应商迟迟不定 | M2 登录拖死 | 先抽象适配层,优先接一个主供应商 |
| 审核后台缺席 | 点评无法上线 | M2 范围内强制带最小审核闭环 |
| 图片链路后置 | 点评功能半残 | `M1` 末尾就把对象存储测试环境接好 |

## 1. 文档目标

这份文档解决的是“线上真炸了以后怎么办”。

不是“理想状态下系统应该很稳定”这种废话，而是:

- 谁先接警
- 谁拍板降级/回滚
- 谁去看日志、回调、数据库、网关
- 多久必须响应
- 事后怎么复盘

线上故障最怕的不是一开始就大爆炸，而是大家都在群里问“这谁负责”“要不要回滚”“我这边看着像网络问题”。这份手册就是拿来堵这种烂场面的。

## 2. 值班角色

### 2.1 最小值班编制

一期建议至少有这几个角色:

| 角色 | 职责 |
|---|---|
| 值班主负责人(On-call Commander) | 统一协调、拍板升级和回滚 |
| 后端值班 | 接口、回调、任务、数据库排查 |
| 前端值班 | Web/管理端/商户端页面与网关排查 |
| 运维值班 | 发布、环境、资源、网络、监控、证书 |
| 业务接口人 | 产品/运营确认影响范围、对外口径 |

### 2.2 角色原则

- 一次事故只能有一个总负责人,别五个人同时指挥。
- 技术排查和业务沟通分开,别让写代码的还得一边安抚群消息。
- 回滚权限要提前约定,不能到时候现问。

## 3. 故障等级

| 级别 | 定义 | 例子 | 响应时限 |
|---|---|---|---|
| P0 | 全站或核心交易/登录不可用,存在严重数据/安全风险 | 登录大面积失败、支付重复入账、数据串区 | 5 分钟内响应 |
| P1 | 重要功能异常,影响大量用户,但有绕路 | 点评无法提交、审核中心积压、验证码失败率飙升 | 10 分钟内响应 |
| P2 | 局部功能或部分区域异常 | EU 区短信失败、某城市 Banner 不显示 | 30 分钟内响应 |
| P3 | 低影响问题或体验问题 | 文案错、样式错、偶发慢请求 | 工作时段处理 |

## 4. 响应流程

### 4.1 标准流程

1. 接警:监控、告警、客服、运营、用户反馈任一渠道触发。
2. 定级:由值班主负责人按影响范围判定 `P0-P3`。
3. 拉群:创建事故群或进入固定事故频道。
4. 止血:优先恢复服务,不是先写一篇分析作文。
5. 排查:按分工查日志、监控、回调、数据库、资源。
6. 决策:降级、限流、切流、回滚、关闭入口等。
7. 恢复:核心功能恢复后持续观察。
8. 复盘:24 小时内补故障记录和改进项。

### 4.2 P0 额外动作

- 立即通知技术负责人和业务负责人。
- 暂停一切非必要发布。
- 必要时关闭高风险入口:
  - 登录/注册
  - 支付下单
  - 预订提交
  - 导出/删除任务创建

## 5. 接警渠道

建议至少接这几类告警:

- 应用监控告警
- 数据库告警
- Redis 告警
- 回调失败率告警
- 验证码失败率告警
- WebSocket 建连异常告警
- 隐私任务失败告警
- 证书/域名到期告警

## 6. 常见故障场景与首查动作

### 6.1 登录/验证码异常

现象:

- 登录失败率飙升
- 验证码发送失败或大量超时

首查动作:

- [ ] 看短信/邮件供应商控制台状态
- [ ] 看 `429` 是否异常增多
- [ ] 看验证码服务配置是否变更
- [ ] 看 Redis 限流键是否误伤
- [ ] 看第三方模板/签名是否失效

止血动作:

- 切备用验证码通道或仅保留邮箱/手机一种登录方式
- 临时放宽部分限流阈值前,先确认不是攻击
- 登录入口加维护提示

### 6.2 支付回调异常

现象:

- 回调失败率升高
- 支付成功但订单未更新
- 重复入账风险

首查动作:

- [ ] 看支付回调日志和验签错误
- [ ] 看回调地址是否被改
- [ ] 看 `idempotency_record` 命中情况
- [ ] 抽查 `orderNo/channelTxn/amount`
- [ ] 看证书或密钥是否过期/错误

止血动作:

- 暂停高风险支付入口
- 临时改为“支付结果待确认”
- 开启补偿任务或人工对账
- 必要时回滚最近支付相关发布

### 6.3 预订状态机异常

现象:

- 预订无法提交
- 状态跳错
- 爽约/到店批量异常

首查动作:

- [ ] 查 `reservation` 与 `reservation_change_log`
- [ ] 查幂等键是否重复使用
- [ ] 查时段容量配置 `reservation_slot`
- [ ] 查异步任务是否积压

止血动作:

- 关闭自动确认,改人工确认
- 暂时关闭改期入口
- 对异常记录手工修正并登记审计

### 6.4 WebSocket/通知异常

现象:

- 大量断连
- 建连失败
- 在线通知延迟明显

首查动作:

- [ ] 看网关连接数、断开率、心跳超时
- [ ] 看 `ws/ticket` 接口成功率
- [ ] 看 TLS/证书状态
- [ ] 看 Redis 或登录态依赖是否异常

止血动作:

- 前端降级为轮询/补拉未读
- 临时缩减广播类消息
- 扩容或重启网关节点

### 6.5 隐私导出/删除任务异常

现象:

- 导出任务堆积
- 删除任务执行失败
- 导出链接失效异常

首查动作:

- [ ] 查 `privacy_export_task` / `privacy_delete_task`
- [ ] 查对象存储权限与导出包生成日志
- [ ] 查定时任务、异步队列积压
- [ ] 查冷静期执行器与设备清理逻辑

止血动作:

- 暂停新建导出/删除任务入口
- 对已创建任务逐条补偿
- 通知业务方延迟处理

## 8. 升级路径

### 8.1 P0

- 值班主负责人
- 技术负责人
- 业务负责人
- 运维负责人

### 8.2 P1

- 值班主负责人
- 对应模块负责人
- 必要时通知业务负责人

### 8.3 什么时候必须升级

- 超过响应时限还没控制住
- 影响范围扩大
- 涉及支付、隐私、安全、数据串区
- 需要回滚或切流

## 9. 降级策略

一期建议把这些降级手段提前准备:

- 登录验证码:邮箱/短信单通道保底
- 点评/帖子:关闭发帖发评入口,保留浏览
- 支付:关闭新下单,保留已支付查询
- 预订:关闭创建/改期入口,保留查询
- WebSocket:降级成未读轮询
- 隐私中心:暂停创建任务,保留查询

## 10. 回滚规则

### 10.1 必须回滚的情况

- 发布后立即引发 P0
- 数据错乱且无法快速止血
- 支付/回调异常率显著升高
- 登录/验证码大面积失效

### 10.2 回滚步骤

1. 确认回滚版本。
2. 暂停继续放量。
3. 回滚应用版本。
4. 回滚配置。
5. 必要时执行数据补偿或数据库回退脚本。
6. 冒烟验证核心链路。

### 10.3 回滚后必须检查

- 首页/列表/详情
- 登录/验证码
- 点评/审核
- 支付回调
- 预订查询
- WebSocket 建连

## 12. 复盘要求

### 12.1 时间要求

- `P0/P1` 事故: `24` 小时内出初版复盘
- `P2` 事故: `3` 个工作日内补记录

### 12.2 复盘必须回答的问题

- 为什么告警没更早发现
- 为什么会扩散
- 为什么没被测试挡住
- 为什么值班时排查效率低
- 后续如何避免再来一次

### 12.3 复盘输出

- 根因
- 时间线
- 影响范围
- 止血动作
- 永久修复项
- 截止日期
- Owner

# 大众点评(仿)项目 · 自动化测试与发布流水线说明

> 版本:v0.1
> 最后更新:2026-07-12
> 配套文档:`M1-M2实施计划与验收清单.md`、`测试清单与验收用例.md`、`上线准备清单.md`、`值班与故障响应手册.md`

## 1. 文档目标

这份文档解决的不是“怎么写代码”,而是“代码写完以后,怎么靠流水线把质量、发布、回滚这几件事卡住”。

前面文档已经把业务、接口、库表、测试范围讲清楚了。现在要补的是:

- 提交代码后自动校验什么
- 合并后自动发到哪个环境
- 上线前后自动跑哪些冒烟
- 冒烟挂了以后怎么自动止血或人工回滚

别再指望上线靠手点、回滚靠记忆、冒烟靠缘分。那不叫流程,那叫碰运气。

## 2. 设计原则

- **同一套脚本本地和 CI 共用**:别本地一套命令、流水线一套命令,最后谁都对不上。
- **先快反馈,再慢校验**:格式、静态检查、单测先跑;集成、回归、部署往后排。
- **环境递进发布**:`dev -> test -> pre -> prod`,别跳级乱发。
- **区域分批放量**:`CN`、`EU` 独立发布、独立验收、独立回滚。
- **数据库迁移单独管控**:发布应用和改表不是一回事,别一锅端。
- **发布必须可审计**:谁发的、发到哪、哪个版本、谁批的、冒烟谁看的,都得留痕。

## 3. 流水线触发矩阵

| 触发场景 | 目标环境 | 必跑阶段 | 是否需要人工审批 |
|---|---|---|---|
| 功能分支 `PR/MR` | 临时校验环境或不部署 | 静态检查、单测、集成测试、构建验证 | 否 |
| 合并到主干 | `test` | 静态检查、单测、集成测试、构建、测试环境部署、冒烟 | 否 |
| 发布分支或 `release tag` | `pre` | 全量校验、预发部署、预发冒烟、回归入口 | 是 |
| 生产发布 | `prod` | 生产部署、生产冒烟、观察期监控 | 是 |
| 定时夜跑 | `test/pre` | 回归集、接口契约、关键 E2E | 否 |
| 手工回滚 | `test/pre/prod` | 回滚、回滚后冒烟、事故记录 | 是 |

补充建议:

- 如果代码托管平台还没定,`GitHub Actions / GitLab CI / Jenkins` 都行,别在工具选型上磨半个月。
- 一期最重要的是把阶段和闸门定住,不是追求花里胡哨的流水线 UI。

## 4. 推荐流水线阶段

### 4.1 Stage A:静态检查

每次提交至少跑这些:

- 代码格式化检查
- Lint / 规范扫描
- 类型检查或基础语义检查
- 依赖漏洞扫描
- 密钥/凭证泄漏扫描
- OpenAPI、枚举、迁移脚本命名规则检查

建议统一脚本入口:

- `scripts/ci/lint`
- `scripts/ci/typecheck`
- `scripts/ci/secret-scan`
- 当前已落地的最小入口:
- `scripts/ci/verify-all.ps1`:本地和 CI 共用的总验证入口,默认跑后端测试、`web` 测试 / 构建、`admin-web` 与 `merchant-web` 验证;带 `-IncludeFlutter` 时追加 `app` 的 `flutter pub get`、`flutter test`、`flutter analyze` 和 Web 构建,带 `-IncludeMysqlSmoke` 时追加真实 MySQL 冒烟,带 `-IncludeStorageSmoke` 时追加 S3 兼容对象存储真上传冒烟,带 `-IncludeBrowserSmoke` / `-IncludeBrowserE2E` 时追加浏览器冒烟 / 真实关键链路 E2E。
  - `scripts/ci/mysql-smoke.ps1`:默认只对本轮新建数据库直接导入 `sql/mysql/01_schema.sql` 和 `02_seed_data.sql`,打包并直接托管后端 jar,轮询 `/actuator/health`,并冒烟 C 端浏览、管理端登录、B 端探活；当 `APP_STATE_STORE_PROVIDER=redis` 时还会补验验证码限流和 `Idempotency-Key` Redis 路径。该脚本已于 `2026-07-12` 用临时 `MySQL 8` 实例 (`127.0.0.1:13306`) 在当前机器实跑通过。
  - `scripts/ci/storage-smoke.ps1`:直接托管 H2 后端,以 `APP_FILE_STORAGE_PROVIDER=s3` 对接 S3 兼容端点,密码登录演示账号后调用 `POST /api/c/v1/files/upload` 真上传 PNG,并校验返回 URL 前缀与收尾清理。
  - `scripts/ci/browser-smoke.ps1`:直接托管 `web` / `admin-web` 的 Vite dev server,并运行 `web/e2e/browser-smoke.spec.ts`,跑最小浏览器冒烟,覆盖首页、商户列表、游客写点评登录拦截、管理端登录页。
  - `scripts/ci/browser-e2e.ps1`:直接托管 H2 后端、`web` 和 `admin-web`,并运行 `web/e2e/real-backend-flow.spec.ts`,跑登录、写点评真实图片上传、成长值流水页、提交点评、管理端审核通过、手机号绑定、改密码后重新登录、后台成功导入、公开点评可见、点赞、评论、举报的真实关键链路。
  - `scripts/ci/package-release.ps1`:构建后端 jar、`web` / `admin-web` dist,并组装版本化 release bundle。
  - `scripts/ci/deploy-release.ps1`:通过 SSH / SCP 上传 release bundle、切换远端 `current`、重启服务并可选执行 smoke URL。
  - `scripts/ci/rollback-release.ps1`:回滚到指定或上一个版本,重启服务并可选执行回滚后 smoke。
- `.github/workflows/ci.yml`:GitHub Actions 校验,使用 MySQL 8 / Redis 7 / MinIO / Elasticsearch service,安装 Playwright Chromium 与 stable Flutter,并通过 `verify-all.ps1` 同时执行后端、Web、管理端、商户端、Flutter 和各 smoke gate。
  - `.github/workflows/nightly.yml`:夜跑回归入口,会在同一套 MySQL / Redis / MinIO service 下追加 `-IncludeBrowserE2E`。
  - `.github/workflows/release.yml`:CI 成功后自动部署 `test`,并保留 `pre/prod` 手工发版入口。
  - `.github/workflows/rollback.yml`:保留 `test/pre/prod` 手工回滚入口。

先把入口名字定住,后面你用 Python、Node 还是别的命令实现都行。别项目还没起,命令先打成一地鸡毛。

### 4.2 Stage B:自动化测试

按层次拆成三类:

#### 1. 单元测试

- 核心领域服务
- 状态机流转
- 评分聚合
- 幂等与限流
- 区域上下文解析

#### 2. 集成测试

- MySQL / Redis 联动
- 对象存储签名上传
- 验证码发送适配层
- 审核任务生成与处理
- 支付/回调验签
- 隐私导出/删除任务执行

#### 3. 端到端冒烟 / 关键链路测试

- 首页 -> 列表 -> 详情
- 游客拦截 -> 登录 -> 恢复原操作
- 写点评 -> 审核 -> 前台可见
- `ws/ticket` -> 建连 -> 心跳
- 支付回调幂等
- 删除任务创建 -> 撤销

## 5. 自动化覆盖策略

### 5.1 M1 必须自动化的内容

| 模块 | 最低要求 |
|---|---|
| 浏览链路 | 首页、列表、详情至少各 `1` 条冒烟 |
| 区域隔离 | `X-Region=CN/EU` 至少各 `1` 条接口校验 |
| 导入链路 | 导入成功、导入失败各 `1` 条集成测试 |
| 公共底座 | 统一错误响应、`traceId`、参数校验要有自动校验 |

### 5.2 M2 必须自动化的内容

| 模块 | 最低要求 |
|---|---|
| 认证鉴权 | 邮箱验证码、手机验证码、refresh、限流 |
| 游客拦截 | 登录前拦截、登录后恢复 |
| 点评体系 | 提交、编辑重审、审核通过、驳回 |
| 图片上传 | 合法图片、非法格式、超限 |
| 审核闭环 | `audit_task` 生成与处理 |

### 5.3 M3+ 预留自动化包

虽然这些能力不是现在立刻上线,但测试位子得先留好:

- 支付回调验签与幂等
- 预订状态机
- WebSocket 建连、重连、踢下线
- 隐私中心导出/删除
- 运营配置变更后的回归

### 5.4 建议闸门

一期别把覆盖率 KPI 玩成形式主义,但也不能一点闸门没有。建议先定这套:

| 指标 | 建议闸门 |
|---|---|
| 核心领域模块单测覆盖 | 不低于 `70%` |
| 新增代码变更行覆盖 | 不低于 `80%` |
| P0 接口契约测试 | `100%` 通过 |
| 发布后冒烟 | `100%` 通过 |

这不是终极真理,但比“看起来差不多”强得多。

## 6. 构建与制品管理

建议把流水线产物拆成三类:

| 产物 | 说明 |
|---|---|
| 后端制品 | 镜像或可部署包 |
| 前端制品 | 静态资源包或镜像 |
| 数据库迁移包 | 迁移脚本、校验脚本、回退脚本 |

要求:

- 每次构建产物都带版本号、提交哈希、构建时间。
- 数据库迁移必须支持 `dry-run`。
- 生产环境部署只允许使用流水线产出的制品,不允许手工在服务器上现编现改。

## 7. 环境发布策略

| 环境 | 用途 | 发布方式 | 要求 |
|---|---|---|---|
| `dev` | 开发自测 | 开发手工或分支自动发 | 可快,但不能脏到没人敢用 |
| `test` | 联调测试 | 合并主干自动发 | 每次发版自动跑冒烟 |
| `pre` | 预发验证 | 发布候选版本手工审批发 | 配置尽量贴近生产 |
| `prod` | 生产 | 手工审批 + 分批放量 | 发版后必须观察监控 |

补充规则:

- `CN`、`EU` 分区域独立配置与发布。
- 配置中心或环境变量必须按环境隔离,别把 `prod` 的密钥塞进 `test`。
- 支付、短信、邮件、审核这类第三方凭证要走密钥管理,别写死在仓库和流水线 YAML 里。

## 8. 数据库迁移规则

发布最容易翻车的地方,很多时候不是应用代码,而是你改表改得跟炸鱼似的。建议强制执行:

1. 迁移脚本提前在 `test/pre` 做 `dry-run`。
2. 生产发布前先备份或确认快照策略。
3. 危险操作拆成 `expand -> migrate -> contract` 三步。
4. 任何删字段、改枚举、批量回填都必须带回退或补偿说明。
5. 数据迁移失败时,优先停在兼容状态,别应用和数据库一起半死不活。

## 9. 发布后自动冒烟

### 9.1 `test` 环境冒烟

- 首页可访问
- 商户列表和详情接口返回 `200`
- 区域头有效
- 管理端登录最小可用

### 9.2 `pre` 环境冒烟

- `M1/M2` 所有 P0 链路跑通
- 第三方测试回调至少演练一次
- 图片上传可用
- 审核任务和后台处理可用

### 9.3 `prod` 环境冒烟

- 首页/列表/详情
- 登录验证码
- 点评提交入口或只读校验
- 关键回调健康检查
- 监控、日志、告警正常入流

生产冒烟别贪多,要快、稳、可重复。你上线后 10 分钟还在手工点一堆边角页面,那说明自动化设计有毛病。

## 10. 回滚策略

### 10.1 触发条件

满足下面任一条件,要么自动回滚,要么立刻走人工回滚审批:

- 发布后核心冒烟失败
- 错误率明显升高并持续
- 登录/支付/预订/隐私任务出现大面积失败
- 区域串数或严重数据错乱
- 第三方回调验签持续异常

### 10.2 回滚动作

| 层级 | 回滚动作 |
|---|---|
| 应用层 | 回滚到上一个稳定版本 |
| 配置层 | 回滚最近一次配置变更 |
| 数据层 | 执行回退脚本或补偿任务 |
| 流量层 | 暂停放量、切回旧实例或关闭高风险入口 |

### 10.3 回滚后必须补的动作

- 自动重跑核心冒烟
- 检查监控是否恢复
- 在事故群补回滚结果
- 补 `事故记录字段模板`

## 11. 监控与告警联动

流水线不是只管发版,还得和监控接上。建议至少联这些信号:

- 应用健康检查失败
- `5xx` 错误率升高
- 验证码失败率升高
- 审核任务积压
- 支付回调失败率升高
- `ws/ticket` 或建连失败率升高
- 隐私导出/删除任务积压

发布后观察期建议:

- `test`: `10-15` 分钟
- `pre`: `15-30` 分钟
- `prod`: `30-60` 分钟

## 12. 建议的流水线任务名

为了后面切 CI 平台不返工,任务名建议先统一:

- `lint`
- `typecheck`
- `unit-test`
- `integration-test`
- `contract-test`
- `build-api`
- `build-web`
- `db-migrate-dry-run`
- `deploy-test`
- `deploy-pre`
- `deploy-prod`
- `smoke-test`
- `rollback`

## 13. M1 立刻要落的最小版本

别一上来就想把终极流水线一步到位,先把下面这些落下来:

1. 代码提交必跑静态检查和单测。
2. 合并主干自动部署 `test`。
3. `test` 部署后自动跑浏览链路冒烟。
4. 发布候选版本支持手工审批发 `pre`。
5. 生产发布保留手工审批和回滚入口。

做到这五条,一期就已经不是裸奔了。

截至 `2026-07-12`,上面这五条已经以最小可用形态落到仓库脚本和 GitHub Actions workflow。当前没闭环的不是“有没有脚本”,而是目标环境的 `DEPLOY_*` / SSH secrets、真实云资源凭证和至少一次发布 / 回滚演练。

## 14. 完成判定

当下面几条都满足时,可以认为自动化这块达标:

- 主干合并后自动校验和自动部署 `test` 已跑通。
- `pre/prod` 都有明确审批闸门。
- 发布后冒烟能自动执行并给出结果。
- 回滚路径做过至少一次演练。
- 监控、告警、事故记录能串上发布版本。

目前仓库已经满足脚本 / workflow 入口层面的要求,离真正达标还差真实环境参数和至少一次目标环境发布 / 回滚演练。

## 15. 下一步

这份文档补完后,后面真正要干的不是再开会空谈流程,而是:

1. 把 `DEPLOY_*`、`APP_S3_*`、`APP_STATE_STORE_PROVIDER` 等真实环境参数填进 CI secrets / 目标机配置。
2. 在真实目标环境演练一遍发布、冒烟、回滚,别只停留在仓库脚本能看。
3. 把执行人、审批人、值班人填到 `执行版联系人与台账模板.md`。
